### 현재 상태 연구 (코드 직접 감사 — sources: src/rider_server/admin/)

**런타임/스택**
- FastAPI + Jinja2 + HTMX 2 (CDN), 빌드 0. 단일 풀페이지 `dashboard.html`(34KB, 인라인 `<style>`) + 6개 부분 템플릿.
- 스타일 = darkmatter 테마(tweakcn `darkmatter.json` 유래). oklch 토큰, 모노 타이포(Geist Mono / JetBrains Mono), 다크 우선, radius 0.75rem, 은은한 도트 텍스처. 라이트/다크 토글(localStorage).
- 데이터: 읽기 전용 `DashboardService` (심각도 합성은 순수 함수 `severity.py`). 쓰기는 별도 action/CRUD 라우터(OPERATOR↑ 권한 + audit).
- 라이브성: 대상/Agent/채널/인증필요 fragment 가 30초 HTMX polling.

**IA — 현재 (단일 롱스크롤)**
- 상단: 앱바(브랜드 · tenant pill · LIVE pill · 갱신시각 · 테마토글) + 섹션 네비(앵커 6개).
- 개요(overview): 상태 배너(ok/warn/crit, attention 합산) + KPI 5개(모니터링 대상/Agent 온라인/인증 필요/Kakao lag/Telegram 오류).
- ① 수동 운영 액션: 구독 suspend/resume, dispatch dispose(DISCARD/RESUME), job retry, agent assign — **전부 raw id 텍스트 입력**.
- ② 엔티티 관리(CRUD): 고객/플랫폼계정/모니터링대상/메신저채널/전송규칙 — 생성 폼 + **편집은 raw id 입력**. secret 은 `*_ref` 핸들만.
- ③ 대상 상태·심각도: 표(심각도 내림차순 정렬). 행 액션 버튼: 활성/비활성/test-crawl/dry-run/인증확인.
- ④ Agent fleet: 표(online/offline·버전·heartbeat·현재 job·capability).
- ⑤ 채널 상태: Kakao queue lag · Telegram 오류 (metric 타일 2개, 의미 혼합 금지).
- ⑥ 인증 필요 대상: 표(tenant·target·profile·reason 코드).

**데이터 필드(실측 — DTO 기준)**
- TargetRow: target_id, tenant_id, name, center_name, platform, interval_minutes, last_success_at(dt|None), last_delivery_at(dt|None), last_failure_code(str|None), severity(NORMAL/WARNING/CRITICAL/STOPPED).
- AgentRow: agent_id, name, version, last_heartbeat_at, online(bool), current_job_type, capabilities(tuple).
- ChannelHealthRow: kakao_queue_lag_seconds, telegram_error_count.
- AuthRequiredRow: tenant_id, target_id|None, profile_id|None, reason(machine code).

### 발견한 UX 문제 (가설 — 사용자 검증 필요)
1. **raw id 타이핑이 도처에** — 구독/dispatch/job/agent 액션과 엔티티 편집 전부 UUID 텍스트 직접 입력. 가장 큰 상처. 오조작·오발송 위험, 개발자 중심.
2. **요약→실제 항목 단절** — KPI/배너가 숫자만 보여주고 클릭으로 해당 항목(위험 대상/offline agent/인증필요)으로 내려가지 못함. drill-down 부재.
3. **관측과 변경이 한 벽에 뒤섞임** — 상단에 거대한 액션·CRUD 폼이 먼저 오고 정작 모니터링 데이터(③④⑤⑥)가 아래로 밀림. "지금 무엇이 문제인가"가 한눈에 안 옴.
4. **시각이 raw datetime** — last_success/heartbeat 가 절대시각 문자열. 운영자는 "3분 전" 상대시간 + 신선도 색이 필요.
5. **실패 사유가 기계 코드** — last_failure_code/auth reason 이 코드값 노출. 사람이 디코딩해야 함.
6. **엔티티 목록이 빈약한 3열 표**(id/summary/state) — 정보 향(scent) 낮음.
7. **빈/로딩/오류 상태 빈약** — "항목이 없습니다" 수준. 인증필요 0건일 때의 안심 메시지, 부분 갱신 실패 표시 등 부재.
8. **모바일/온콜** — 640px 분기는 있으나 밀집 표 + raw id 폼은 폰에서 실사용 어려움.

## 2026-06-15 · 세션 1 — 코칭 5단계: 상태·인터랙션·접근성·말투 (D14, 전부 채택)

### 상태 패턴
- **전부 정상** → 빈 표 금지, "🟢 N곳 모두 정상 · 지금 조치할 항목 없음" 안심 메시지.
- **필터 결과 0** → "조건에 맞는 업체 없음 — 필터 조정" + 필터 초기화.
- **첫 로딩** → 스켈레톤 행(레이아웃 유지, 깜빡임 방지).
