# Story 1.4: 기존 2탭 수동 회귀 시나리오 문서화

## Story

As a 운영자,
I want 현재 활성 **배민 1탭·쿠팡 1탭**에 대한 수동 회귀·dry-run(실발송 없는) 비교 절차를 `docs/qa/` 런북으로 문서화하고, 그 산출물이 secret 없이 보존됐는지 자동 회귀 가드 테스트로 고정하고,
so that 리팩토링 각 단계 후 동일 절차를 반복해 **기존 배민/쿠팡 수집·렌더링·전송 동작이 보존됐는지**를 사람이 검증할 수 있고, 향후 Epic 3 신규 경로의 dry-run 비교(FR-3)가 참조할 기준선과 비교 방법이 미리 마련된다.

> **이 스토리의 성격(중요 — 1.1/1.2와 같은 "문서·산출물 스토리", 1.3과 다른 코드 스토리 아님).** Epic 1(기준선 안전망, P0)의 네 번째 스토리이며 spec 단계 **P0-05**를 구현한다. 산출물은 **① 수동 회귀 런북(`docs/qa/`), ② 실발송 없는 dry-run 기준선 기록(`docs/qa/`), ③ 그 산출물을 지키는 회귀 가드 테스트(`tests/`)** 다. **제품 코드(`src/rider_crawl/`)는 변경하지 않는다** — 기존 `run_once`/`render_current_screen_message`/메신저 경계를 **그대로 사용·문서화**할 뿐 끼워 넣거나 고치지 않는다. 현재 HEAD(`a820a85`) 기준선 전체 스위트(Story 1.3 시점 **499 passed** — 참고값, 복사 금지·실측 확인)는 그대로 통과해야 하고, 신규 가드 테스트 케이스 수만큼만 passed가 증가하는 게 정상이다(NFR-20).
>
> **실발송 절대 금지.** 이 스토리의 모든 dry-run/테스트 절차는 실제 Telegram/Kakao 발송을 **하지 않는다**(FR-3, NFR-1 fail-closed). 발송이 필요한 "test send" 절차는 운영자가 **지정 테스트 채널에 한해 수동으로만** 수행하도록 런북에 격리해 기술한다(자동화/CI 발송 금지).

## Acceptance Criteria

**AC1 — 4개 수동 회귀 절차 + 기대 결과 문서화 (P0-05, FR-1)**
1. **Given** 현재 활성 배민 대상 1개와 쿠팡 대상 1개가 있을 때 **When** 수동 회귀 시나리오를 `docs/qa/`(런북)에 문서화하면 **Then** **① 배민 run, ② 쿠팡 run, ③ Telegram 테스트 전송, ④ Kakao 테스트 전송** 절차가 각각 운영자가 그대로 따라할 수 있는 **단계별(번호/체크박스)** 형태로 기록된다(P0-05 "Baemin run, Coupang run, Telegram/Kakao test procedures are documented").
2. **And** 네 절차 각각에 **기대 결과**가 함께 명시된다 — 최소: (a) 수집 성공의 판정 기준(예: 예외 없이 `RunResult`가 반환되고 `message`가 비어 있지 않음), (b) 렌더링된 메시지의 **형태**(배민: `[실시간 실적봇]` 헤더 + `[<센터명/탭라벨>]` + `… 기준` 타임스탬프 + `오전오후피크/오후논피크/저녁피크/저녁논피크` 4줄 + 선택적 `거절율`; 쿠팡: `[실시간 실적봇]` 헤더 + 피크 대시보드 실적 형태). [Source: src/rider_crawl/message.py `render_current_screen_message`(35)]
3. **And** Kakao 테스트 전송 절차는 기존 `docs/kakao-verification-checklist.md`(정확한 채팅방명 일치·안전 실패 정책)를 **재사용·교차참조**하고 동일 검증 로직을 중복 서술하지 않는다(wheel 재발명 금지).

**AC2 — 실발송 없는 dry-run 기준선 1회 캡처 + 보존 (FR-3)**
4. **Given** 기준선 수집/렌더링 결과를 남겨야 할 때 **When** 대표 배민/쿠팡 대상의 **수집·렌더링 dry-run을 1회 실행**(실발송 OFF)하면 **Then** 그 결과가 `docs/qa/`의 dry-run 기준선 기록에 저장된다 — 저장 항목은 최소 **대상별 렌더링 메시지의 sha256 해시**(= `run_once`가 이미 계산하는 `RunResult.message_hash`)와 **sanitized 메시지 스켈레톤**(라벨/구조 보존, 숫자·센터명 등 변동/식별 값은 마스킹)와 캡처 메타데이터(일시 KST, 플랫폼, 탭 라벨, 실행 방식)다. [Source: src/rider_crawl/app.py run_once(41·51)]
5. **And** 이 dry-run은 **실제 Telegram/Kakao 발송을 하지 않는다**(FR-3) — 비발송은 `send_enabled=False` 경로(`run_once`가 `sent=False, skipped=False`로 반환, sender 미호출) 또는 부작용 없는 fake `send_message` 주입으로 보장하며, 런북에 "실발송 OFF 확인" 단계를 명시한다. [Source: src/rider_crawl/app.py run_once(46-51), config.py send_enabled default False(94)]
6. **And** 기준선 기록 파일과 런북 어디에도 **실제 secret(토큰/비밀번호/OTP/chat_id/전화번호)이나 마스킹되지 않은 운영 식별자 원문**이 남지 않는다 — 커밋 전 Story 1.3의 `rider_crawl.redaction.redact(..., mask_operational_ids=True)`를 통과시켜 sanitize한다(NFR-5, ADD-15). [Source: src/rider_crawl/redaction.py, 1-3 스토리]

**AC3 — 재실행 비교 방법 명문화 (FR-3, NFR-24)**
7. **Given** 향후 단계에서 메시지 차이를 비교해야 할 때 **When** 문서화된 절차를 다시 수행하면 **Then** 기준선 메시지와 신규 메시지를 비교할 수 있는 **저장 위치와 비교 방법**이 절차에 포함된다 — 최소: (a) 동일 입력(동일 snapshot/대상)일 때 sha256 해시 일치로 형태 동일성 확인, (b) 실시간 수치가 매 실행 달라지는 라이브 dry-run은 **형태/라벨 단위 비교**(숫자 제외)로 회귀를 판정한다는 점을 명시.
8. **And** 결정적(deterministic) **렌더 형식 회귀**는 이미 기존 골든 테스트 `tests/test_message.py`·`tests/test_coupang_message.py`(고정 snapshot→정확한 메시지 단언, Story 1.2 pytest 기준선에 포함)가 잠그고 있음을 비교 방법 문서에 연결하고, **그 골든 테스트를 중복 재작성하지 않는다**. 라이브 dry-run은 "수집→렌더 end-to-end가 여전히 동작하고 메시지가 well-formed이며 실발송이 없음"을 확인하는 운영자 스모크임을 구분해 기술한다.
9. **And** old path(기존 UI/`run_once`)와 new path(Epic 3 신규 경로)의 **동시 실제 전송을 막는 cutover 규칙(NFR-24)**의 전제가 "dry-run은 실발송 없음 + 승인 후에만 활성화"임을 비교 절차에 한 줄로 못박는다(FR-3: 차이 발생 시 자동 활성화 금지).

**AC4 — 산출물 회귀 가드 테스트 (NFR-20, NFR-5, QA 자동화)**
10. **Given** 이 스토리가 수동 절차를 자동 회귀로 고정해야 할 때 **When** `tests/`에 가드 테스트(예: `tests/test_manual_regression_runbook.py`)를 추가하면 **Then** 런북과 dry-run 기준선 파일이 **존재**하고, AC1의 4개 절차 섹션과 각 기대 결과·AC3의 비교 방법 섹션이 **모두 포함**됐는지 단언한다.
11. **And** 그 테스트는 두 산출물(및 함께 커밋되는 텍스트)에 **실제 텔레그램 봇 토큰 형태(`\d{6,}:[A-Za-z0-9_-]{30,}`)·full email·실제 휴대폰 형태·`chat_id=<digits>` 평문이 없음**을 **정규식 패턴 검사**로 단언한다(`test_baseline_artifacts.py`와 동일 가드 철학). 외부 브라우저/Telegram/Kakao/Gmail을 호출하지 않는 **순수 파일 읽기 테스트**다(project-context 테스트 규칙). [Source: tests/test_baseline_artifacts.py]

### dry-run 실행 경로 정본 (모호함 제거 — 코드 근거)

- `run_once(config, *, crawl_snapshot=None, send_message=None) -> RunResult`는 수집→렌더→(중복 확인)→전송을 묶는 경계다. 분기:
  - `config.send_only_on_change and _is_duplicate(...)` → `sent=False, skipped=True`.
  - `config.send_enabled` 참 → sender 호출 후 `sent=True`.
  - **그 외(= `send_enabled=False`) → sender 미호출, `sent=False, skipped=False`로 `message`/`message_hash` 반환.** ← **이게 dry-run의 정본 경로다.** [Source: app.py(43-51)]
- `RunResult`(frozen dataclass): `message`, `sent`, `skipped`, `message_hash`. `message_hash = hashlib.sha256(message.encode("utf-8")).hexdigest()` — 기준선 비교 키로 그대로 쓴다(`send_only_on_change` 중복 판정도 같은 해시를 쓴다 → 의미 일관). [Source: app.py(15-20·41)]
- **활성 탭은 UI per-tab 설정**(`runtime/state/ui_settings.json`)에 있다. UI `1회 실행`은 `run_once_clicked → _run_once_background`가 `settings.to_app_config(...)`로 탭 설정을 써서 `run_once`를 호출한다 → 활성 탭 dry-run의 정본 경로. [Source: ui.py(533·613-616)]
- **CLI `--once` caveat(흔한 실수 지점):** `python -m rider_crawl --once` → `run_cli_once()` → `run_once(AppConfig.from_env())`. **`.env`만 읽고 UI 탭을 읽지 않는다.** 활성 탭과 동일 결과를 보장하지 않으므로 "활성 탭 회귀"용으로는 UI 경로를 기본으로 하고, CLI는 `.env`가 탭과 동일하게 구성된 보조 수단으로만 안내한다. [Source: ui.py(984-997), config.py from_env(94)]

### secret/식별자 안전 (NFR-5, ADD-15 — 커밋되는 산출물이므로 필수)

- 커밋되는 런북·기준선에 **실제 secret(token/password/OTP/`chat_id`/full phone/full email) 평문 금지.** 모두 placeholder(`<TELEGRAM_BOT_TOKEN>`, `<TELEGRAM_CHAT_ID>`) 또는 가짜값(`010-0000-0000`).
- 렌더 메시지 스켈레톤은 **운영 식별자(센터명 등)까지 마스킹**해서 커밋한다 — Story 1.3의 `redact(text, mask_operational_ids=True)`를 재사용한다(새 마스킹 로직 작성 금지). `*_ref`는 secret 아님(마스킹 대상 아님). [Source: redaction.py, 1-3 AC2 #5·File List]
- dry-run 기준선의 **sha256는 메시지 본문에서 산출한 불투명 해시**라 secret이 아니다(원문 복원 불가) → 그대로 커밋 가능. 본문 스켈레톤만 sanitize하면 된다.
