## Story

As a 운영자,
I want 기존 `ui_settings.json`의 활성 탭을 **① 원본 백업 → ② 활성/비활성 분류(비활성은 보존·자동 활성화 금지) → ③ `tenant_id`·`platform_account_id`·`monitoring_target_id` 발급 → ④ 도메인 모델(Tenant/PlatformAccount/MonitoringTarget) 매핑 → ⑤ `runtime/state/crawlingN` → `targets/<monitoring_target_id>` 상태 복사(원본 미삭제) → ⑥ 기존 `last_message` hash를 DeliveryLog/idempotency seed로 승계** 하는 **순수·결정적 마이그레이션 오케스트레이션**(`src/rider_server/migration/`)으로 실행하고, 각 대상의 진행 상태를 **`MigrationState`(discovered/mapped/dry_run_passed/approved/active/paused/rolled_back)** 상태머신으로 표현하되 **실제 DeliveryRule 활성화·발송은 운영자 승인 전까지 절대 일어나지 않게(fail-closed)** 하고 싶다,
so that 마이그레이션 과정에서 **원본 손실(NFR-18)·중복 발송(NFR-21)·비활성 대상 자동 활성화 없이** 안전하게 ID 기반 모델로 전환되고, 이후 dry-run 렌더 비교(Story 3.8)·DeliveryRule fan-out(Story 3.4)·DeliveryLog 영속(Epic 5)이 **동일한 마이그레이션 산출물(매핑·seed·상태)을 정본으로 소비**한다(ADD-16, FR-31).

> **이 스토리의 성격 — "기존 로컬 설정(`ui_settings.json`/`runtime/state`)을 ID 기반 도메인 모델로 옮기는 순수 마이그레이션 오케스트레이션." 실제 발송도, dry-run 렌더 비교도, DeliveryRule 활성화도, DB/영속도 아니다.** Epic 2는 "도메인 모델·상태(enum) 진화 + `src/rider_crawl` 설정 진화" 에픽이고(epics.md 353-355), 본 스토리는 그 에픽의 **마무리 — 2.1~2.6이 만든 ID 필드·중립 필드·secret_ref·도메인 모델·상태 enum을 실제로 "기존 활성 탭"에 적용해 옮기는 절차**다. **2.5가 명시적으로 본 스토리에 위임했다: `monitoring_target.py`(9-10) "2.1~2.3 UiSettings 중립 필드와의 wiring은 Story 2.7 소유", `domain/__init__.py`(5-6) "wiring은 Epic 5 / Story 2.7".** 본 스토리의 ADD-16/FR-31 deliverable은 **"백업·분류·발급·매핑·상태복사·seed 승계를 결정론적으로 수행하고, 각 대상의 마이그레이션 진행을 상태로 표현하며, 운영자 승인 전 활성화 0을 코드·테스트로 잠그는 것"** 이다. **실제 dry-run 렌더 비교(old vs new)는 Story 3.8(P2 cutover), DeliveryRule fan-out/활성화는 Story 3.4 + Epic 5, DeliveryLog/idempotency 영속·유니크 제약은 Epic 5(P4-02), kill switch/canary는 Epic 5/P6, scheduler 연동은 Story 5.4 소유다.** [Source: epics.md Story 2.7(487-509)·Epic 2(353-355), implementation-contract.md Migration Contract(96-105), src/rider_server/domain/monitoring_target.py(9-10)·__init__.py(5-6), architecture.md(442·507·153-154)]

> **fail-closed가 최상위 안전 원칙(NFR — 오발송·원본손실보다 미발송·보존).** 본 마이그레이션의 존재 이유는 "전환하다 원본을 잃거나, 중복 발송하거나, 멈춘 탭을 깨우지 않기" 다. 따라서 **네 불변식**을 코드와 테스트로 강제한다: ① **원본 미삭제** — 백업·상태 복사는 원본(`ui_settings.json`, `runtime/state/crawlingN`)을 **읽기만** 하고 삭제·이동하지 않는다(NFR-18), ② **비활성 탭 자동 활성화 0** — `performance_url`이 빈 탭은 target 후보가 아니며 ID 발급·상태 복사·매핑 대상에서 제외하고 보존한다, ③ **승인 전 활성화 0** — 마이그레이션은 기본적으로 `MAPPED`에서 멈추고, `APPROVED`→`ACTIVE` 전이는 **운영자 입력(명시 호출)** 없이 자동으로 일어나지 않는다(활성 DeliveryRule도 만들지 않는다), ④ **dedup 연속성** — old `last_message` hash가 신규 경로로 승계돼 마이그레이션 직후 같은 메시지가 중복 발송되지 않는다(NFR-21). [Source: epics.md AC1~AC3(495-509), implementation-contract.md(98-105), project-context.md(86-89·92), architecture.md(45·329·556)]
>
> **secret/식별자 비노출(NFR-5·NFR-8·ADD-15).** 매핑이 만드는 `PlatformAccount`는 자격증명을 **평문이 아니라 `SecretRef`**(`username_ref`/`password_ref`)로만 가리킨다(2.4·2.5 계약). 본 마이그레이션은 **신규 ID-모델 산출물에 평문 secret을 절대 쓰지 않는다**(ADD-15). 단, **원본 백업은 원본을 그대로 보존**하는 롤백 아티팩트이므로(NFR-18) redaction하지 않는다 — ADD-15의 "신규 파일"은 마이그레이션이 **생성하는 ID-모델 파일/매핑**을 뜻하고 원본 백업은 거기에 해당하지 않는다(아래 Dev Notes "백업 충실도 vs ADD-15" 참고). 테스트 fixture·로그·예외 메시지엔 실제 봇 토큰/비밀번호/OTP/`chat_id`/전화/이메일 원문을 넣지 않고 명백한 가짜값만 쓴다. [Source: project-context.md(81·89), src/rider_server/domain/platform_account.py(20-23)·secret_ref.py(16-20), 2-4 스토리, epic-1-retro 액션 A1]

## Acceptance Criteria

**AC1 — 백업 → 활성/비활성 분류(비활성 보존·자동 활성화 금지) → ID 발급 (ADD-16 1~3단계, FR-31, NFR-18)**

1. **Given** 기존 `runtime/state/ui_settings.json`의 `crawlings` 배열이 있을 때 **When** 마이그레이션을 실행하면(ADD-16, FR-31) **Then** **기존 설정이 어떤 변형(ID 발급/secret 흡수)보다 먼저 원본 그대로 백업**되고(주입된 backup 경로로 원자적 복사, 원본 미삭제 — NFR-18), 백업은 마이그레이션 실패 시 롤백 가능한 충실한 사본이다. [Source: epics.md AC1(495-497), implementation-contract.md(98), architecture.md(466 runtime 보존)]
2. **And** `crawlings` 배열의 **탭 중 `performance_url`이 비어 있지 않은 활성 탭만 target 후보로 분류**되고(분류 기준은 `rider_crawl`의 활성 판정 `performance_url.strip()`과 동일 — `ui.active_crawling_settings`/`ui_settings.load_all`(244) 정본), **`performance_url`이 빈 비활성 탭은 보존하되 ID 발급·상태 복사·매핑·활성화 대상에서 제외**된다(자동 활성화 0 — 불변식 ②). `load_all`의 9-slot 패딩으로 생기는 기본 슬롯은 후보가 아니다(파일에 실제 존재한 탭만). [Source: epics.md AC1(498), src/rider_crawl/ui_settings.py(243-247), src/rider_crawl/ui.py(101)]
3. **And** **각 활성 탭에 `tenant_id`·`platform_account_id`·`monitoring_target_id`가 발급**된다 — 발급은 **`UiSettingsStore.load_all`(2.1 `_issue_missing_ids`)을 재사용**하고(반-재발명: 활성 탭에만 안정 `uuid4` 발급 + 영속화, 재로드 시 동일 ID 유지), 본 모듈은 `uuid4`를 직접 호출하지 않는다. `customer_id`(2.1)가 곧 도메인 `tenant_id`이다(같은 불투명 ID). [Source: epics.md AC1(499), implementation-contract.md(100·P1-01 36), src/rider_crawl/ui_settings.py(334-356)]

**AC2 — 상태 폴더 복사(원본 미삭제) + `last_message` hash → DeliveryLog/idempotency seed 승계 (ADD-16 4~5단계, NFR-18·NFR-21)**

4. **Given** 기존 상태 폴더와 중복방지 기록을 승계해야 할 때 **When** 마이그레이션이 상태를 복사하면 **Then** 각 활성 대상의 **`runtime/state/crawling{N}` 폴더 내용이 `runtime/state/targets/<monitoring_target_id>`로 복사**되고(대상 경로는 2.2 `state_subdir=targets/<id>`·`config.state_dir` 정본과 일치), **원본 `crawling{N}`은 삭제·이동되지 않는다**(NFR-18, 불변식 ①). `{N}`은 해당 활성 탭의 1-based 파일 위치(legacy `crawlingN` 순번)다. 대상 폴더가 이미 있으면 원본을 덮어쓰지 않는 안전 병합(없는 파일만 복사)으로 멱등하게 처리한다. [Source: epics.md AC2(501-503), implementation-contract.md(101), src/rider_crawl/config.py(128-134), src/rider_crawl/ui.py(95-104)]
5. **And** **기존 `last_message.<scope_hash>.sha256`의 message hash가 신규 DeliveryLog/idempotency seed로 승계**된다(NFR-21) — (a) 상태 폴더 복사가 `last_message.*.sha256` 파일을 그대로 옮겨 `rider_crawl.app._is_duplicate`가 신규 `targets/<id>` 경로에서도 같은 hash를 읽어 **마이그레이션 직후 같은 메시지의 중복 발송을 막고**(불변식 ④), (b) 추가로 본 마이그레이션은 `MigrationSeed`(값 객체: `monitoring_target_id` + old `message_hash` + `scope_hash`)를 산출해 **Epic 3/5의 DeliveryLog/idempotency가 소비할 seed 정본**을 표현한다(실제 `delivery_logs` 행·유니크 제약은 Epic 5). [Source: epics.md AC2(504), data-api-contract.md(146-156), src/rider_crawl/app.py(54-65), implementation-contract.md(102)]

**AC3 — 마이그레이션 진행 상태머신 + 승인 전 활성화 0 (NFR-22, FR-3 연계)**

6. **Given** 마이그레이션 진행 상태를 추적해야 할 때 **When** 각 대상이 단계를 거치면 **Then** **`MigrationState` enum(`DISCOVERED`/`MAPPED`/`DRY_RUN_PASSED`/`APPROVED`/`ACTIVE`/`PAUSED`/`ROLLED_BACK`)** 으로 진행 상태를 표현할 수 있고(NFR-22, `(str, Enum)`+대문자 정본 — 2.5 enum 컨벤션 계승), 각 대상의 마이그레이션 결과(`TargetMigration`)는 상태·매핑·seed를 담는 frozen 값 객체로 반환된다. 전이는 **결정론적 순수 함수**(`dataclasses.replace`)로, 비활성 탭은 `DISCOVERED`(미활성화·보존)로 기록된다. [Source: epics.md AC3(506-508), architecture.md(153-154), src/rider_server/domain/states.py(1-7 enum 컨벤션)]
7. **And** **실제 DeliveryRule 활성화는 운영자 승인 전에는 일어나지 않는다**(FR-3 연계) — 마이그레이션 `run_migration`은 기본적으로 활성 대상을 **`MAPPED`에서 멈추고**(자동으로 `APPROVED`/`ACTIVE`로 진행하지 않으며 활성 DeliveryRule을 만들지 않는다, 불변식 ③), `APPROVED`/`ACTIVE` 전이는 **운영자 입력을 표현하는 명시 전이 함수**(`approve`/`activate`) 호출로만 가능하다. 유효하지 않은 전이(예: `DISCOVERED`→`ACTIVE` 직행, 승인 없는 활성화)는 **재발송/오활성화를 막기 위해 명확한 예외**(`ValueError`)를 낸다(fail-closed). 실제 dry-run 렌더 비교는 Epic 3(Story 3.8) 소유다. [Source: epics.md AC3(509), implementation-contract.md(103-105), architecture.md(329·556), project-context.md(86-89)]

### 백업 충실도 vs ADD-15 (헷갈리기 쉬운 경계 — 반드시 읽을 것)

- **긴장:** NFR-18은 "원본 보존"(롤백 위해 충실한 백업)을, ADD-15는 "마이그레이션 시 평문 secret이 **신규 파일**로 복사 금지"를 요구한다. legacy 원본이 평문 secret을 담고 있으면 둘이 충돌하는 것처럼 보인다.
- **해소:** **백업은 원본의 충실한 사본(롤백 아티팩트)** 이지 마이그레이션이 **생성하는 신규 ID-모델 산출물이 아니다.** ADD-15의 "신규 파일"은 마이그레이션이 만드는 **새 ID-모델 설정/매핑/target 산출물**을 뜻한다(2.4가 `to_jsonable`로 ref만 남기는 신규 `ui_settings.json`이 그 예). 따라서: **백업은 redaction 없이 원본 그대로**(NFR-18 우선), **신규 매핑(`PlatformAccount`)은 `SecretRef`만**(ADD-15), **상태 복사는 hash만**(secret 없음). 본 마이그레이션은 **새 평문을 단 한 곳에도 만들지 않는다.** (참고: `load_all` 재사용이 원본 `ui_settings.json` 자체의 평문을 2.4 정책대로 store로 흡수해 ref로 만들지만, 그건 2.4 소유 동작이고 본 스토리는 그 동작에 의존만 한다.) [Source: project-context.md(81·89·92), src/rider_crawl/ui_settings.py(273-298·371-379), 2-4 스토리]
