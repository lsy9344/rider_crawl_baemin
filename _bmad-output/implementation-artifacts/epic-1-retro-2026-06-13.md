## 2. 잘된 점 (Successes)

1. **"문서·산출물 스토리 + 순수 파일 읽기 가드 테스트" 패턴의 일관 적용 (5개 중 4개).** 제품 코드 변경 없이도 안전망 산출물을 자동 회귀로 고정해, 산출물이 사라지거나 secret이 섞이면 테스트가 즉시 깨지는 구조를 만들었다. 안전망 에픽의 목표(되돌릴 수 있고 무엇이 바뀌었는지 안다)에 정확히 부합한다.

2. **엄격한 범위 규율 — 스코프 크립 0건.** 모든 스토리가 "문서화일 뿐 구현이 아니다"를 명시하고 retrofit("required change")을 소유 에픽(Epic 2~5)으로 미뤘다. 특히 1.3은 redaction 유틸만 만들고 기존 `sender.py`/`app.py` 경계에 끼워 넣지 않았다(후속 P3/P4 책임으로 명확히 분리).

3. **적대적 리뷰가 "done" 직전에 실제 결함을 잡았다.** 1.3 리뷰에서 `Authorization: Bearer <token>` 자유 텍스트 누출(AC2 위반 — 토큰 본문 평문 잔존)을 발견해 `_AUTH_HEADER_RE` 전체 라인 마스킹으로 수정했다. 리뷰가 형식 검토에 그치지 않고 NFR-5 위반을 실측 차단한 사례.

4. **토대의 즉시 재사용.** 1.3 `redact(..., mask_operational_ids=True)`를 1.4·1.5가 스켈레톤 sanitize에 재사용했고, `docs/qa/` 컨벤션은 1.1→1.5까지, 가드 테스트 패턴·secret 정규식은 1.1→1.4→1.5로 import 재사용됐다. 재발명 없이 누적.

5. **NFR-20(기존 테스트 유지)을 전 구간 무회귀로 충족.** 397→584로 187 케이스가 늘어나는 동안 기준선 통과 테스트가 새로 깨진 적이 없다.

## 3. 어려웠던 점 / 반복 패턴 (Challenges — 무비난, 시스템 관점)

1. **테스트 수치 stale (반복 2회: 1.4 MEDIUM-1, 1.5 동일 유형).** dev-story 시점에 기록한 passed 수치가 이후 QA 자동화 보강으로 낡아, 리뷰가 매번 재측정해 File List·Change Log를 정정했다. 근본 원인은 시스템적이다: `dev-story → QA 보강 → review` 순서에서 중간에 적는 절대 수치는 항상 잠정값이 된다.

2. **secret 누출 near-miss (1.1 ×2, 1.3 ×1).** 명시적 정책에도 불구하고 커밋 대상 산출물에 실제 민감 부분 문자열이 섞였다 — (1.1) 디렉터리 경로명 속 쿠팡 login-id, (1.1) `.env.example` 주석 속 센터명 단편, (1.3) Authorization 헤더 토큰. 모두 작성 시점이 아니라 사후 스캔/리뷰에서 잡혔다. 수동 점검에 의존하는 한 재발 여지가 있다는 신호.

3. **CRLF/LF EOL 노이즈 (1.1).** 추적 텍스트 파일(`.env.example`) 편집 시 파일 전체 EOL이 뒤집혀(LF↔CRLF) 실제 보안 변경 4줄이 50줄 통째 diff에 묻혔다. `.gitattributes` 정규화가 없어 diff 위생 위험이 상존한다.

4. **환경 마찰.** WSL 시스템 `python3`에는 pytest가 없어 반드시 운영 venv `.venv/Scripts/python.exe`로 실행해야 한다. 매 스토리에서 반복 확인된 발끝 걸림(메모리 `dev-env-quirks`에 기록됨).

5. **라이브 캡처 미완 (1.4).** dry-run 기준선의 실측 sha256/스켈레톤이 `<운영자 캡처 필요>` placeholder로 남아 있다. 로그인된 Chrome(CDP)으로 운영자가 채워야 실제 비교 기준선이 된다. (단 실제 dry-run 경로 구현은 Epic 3 책임이라 Epic 2 진입을 막지는 않는다.)

## 8. 핵심 takeaway

1. **안전망은 "산출물 + 그 산출물을 지키는 가드 테스트" 쌍으로 만들 때 실제로 회귀를 잡는다.** Epic 1의 4개 문서 스토리가 모두 이 쌍을 채택했고, 다음 에픽들의 능동 회귀 그물이 됐다.
2. **secret 비노출은 수동 점검이 아니라 자동 게이트여야 한다.** 3건의 near-miss가 모두 사후 발견이었다 → A1.
3. **범위 규율(retrofit을 소유 에픽으로 미루기)이 무회귀의 핵심이었다.** Epic 2는 이 규율을 코드 변경 맥락에서 유지하는 것이 과제다.
