## 1. 에픽 요약과 지표

Epic 3은 강결합된 단일 실행 경계 `app.run_once`를 **CrawlService / MessageRenderService / DispatchService** 세 책임으로 분해하고, 그 위에 정규화 `Snapshot`(fail-closed)·`Message`(안정 hash)·`DeliveryRule` fan-out·`DeliveryLog`+idempotency(crash-after-send 중복 차단)·실패 카테고리/재시도 정책·Telegram 중앙(send-only) 전송·dry-run 비교 후 승인 활성화를 **순수·동기·런타임 미배선**으로 쌓은 P2 에픽이다. Epic 2가 "데이터 모델·설정 진화"였다면 Epic 3는 **"실행 흐름 재배선"** 이다(epic-2-retro 104). 핵심 제약은 단 하나로 일관됐다: **`src/rider_crawl/`는 8개 스토리 전부에서 0줄도 바꾸지 않고**(단방향 import·호환 경로 보존), 모든 신규 의미는 `src/rider_server/`에 additive로 얹는다. Epic 1이 깐 가드/골든 테스트와 Epic 2의 도메인 토대가 이 재배선의 안전망이 됐다.

## 3. 잘된 점 (Successes)

1. **`rider_crawl` 0줄 변경 — 실행 흐름 재배선 에픽에서도 호환 경로 완전 보존.** 8개 스토리가 `run_once`를 포함해 `src/rider_crawl/` 전체를 단 한 줄도 고치지 않고(`git diff -w 4636b0e..HEAD -- src/rider_crawl/` 빈 출력), 신규 서비스가 `rider_crawl`을 **단방향으로만** import했다. 덕분에 "기존 UI 1회 실행 결과 동일"(NFR-20·FR-2)이 자명하게 성립하고, `tests/test_app.py`·parser 골든 테스트가 전 구간 무회귀였다. 의존성 방향 위반 0건은 8/8 스토리의 AST import-엣지 가드로 잠겼다.

2. **NFR-20 전 구간 무회귀 + 결정론 일관.** 786→994로 +208 케이스가 늘고 실행 경계를 통째로 분해하는 동안 기준선 통과 테스트가 새로 깨진 적 **0건**. 모든 비결정성(`datetime.now()`·`uuid4()`·`random`·I/O)을 **호출부 주입 seam**으로 밀어내(3.1~3.8 일관) 게이트·정규화·dedup·backoff·cutover가 100% 재현 가능해졌다. 3.6 backoff는 `min(cap, base·factor^(attempt-1))`로 결정적이고 jitter는 5.4로 위임했다.

3. **"새 필드가 아니라 새 뷰/레코드/단위" + compose-only(재구현 0) 무회귀 패턴의 전수 적용.** 3.1은 `run_once`를 갈아엎지 않고 서비스 경계를 옆에 추가, 3.2는 parser 출력을 `asdict`로 감싸기만, 3.3은 기존 renderer를 재사용(`render_message().text == render_current_screen_message(...)` 바이트 동등), 3.6/3.8은 3.5/2.7의 함수·상태머신을 **compose만** 하고 재구현하지 않았다. Epic 2 회고가 지목한 무회귀 비결이 코드 에픽에서도 정확히 유지됐다.

4. **도메인 vs 서비스 레이어 배치 규율의 일관성.** 계약 테이블을 백킹하는 레코드만 `domain/`에 넣었다(`Snapshot`=9·`Message`=10·`DeliveryLog`=11). 파이프라인 중간 값 객체(`DispatchResult`·`DispatchJob`·`FanoutOutcome`·`RetryDecision`·`TelegramRoute`·`DryRunResult` 등)는 `services/`/`migration/`에 둬 `domain.__all__` 정본 잠금을 불필요하게 흔들지 않았다. 3.4/3.7/3.8은 domain lock 무변경, 3.3/3.5/3.6만 의도적으로 갱신.

5. **fail-closed가 모든 단계에 관통(오발송보다 미발송).** 3.1 단계 실패 비전파 → 3.2 필수값 누락 시 예외(`MissingSnapshotDataError`, 0/기본값 주입 금지) → 3.4 dangling 채널 FK → `UnknownChannelError` → 3.5 insert-then-send(성공 전송 전 dedup key 선확보) → 3.6 `AUTH_REQUIRED`/`TARGET_VALIDATION_FAILURE` → `HELD`(무한 재시도 금지) → 3.7 ambiguous 실패 시 미재전송 기본값 → 3.8 `seed=None`/미승인 차이/동시전송 → 차단. 안전 기본값이 한 스토리도 빠짐없이 적용됐다.

6. **3.7 Telegram 중앙화의 비침습 극대화 + 큐 경합 원천 차단.** legacy `send_telegram_text`를 무변경 재사용하면서 `getUpdates`/`TelegramUpdatePoller`를 **import조차 하지 않도록** 했고(AST 가드 `test_module_is_send_only_no_getupdates_or_poller`로 단언), "같은 봇 토큰 다중 polling 금지"(project-context 규칙)를 구조적으로 보장했다. enum·도메인 모델 추가 0으로 lock 갱신도 0 — Epic 3에서 가장 비침습적 스토리.

## 4. 어려웠던 점 / 반복 패턴 (Challenges — 무비난, 시스템 관점)

1. **테스트 수치 stale — 8/8 전수 재발 (Epic 1 A2 → Epic 2 → Epic 3 직접 연장).** 모든 스토리에서 dev-story 시점 기록 수치가 이후 qa-generate-e2e 케이스 추가로 낡았고, 리뷰가 매번 재측정해 정정(MEDIUM 8건 전부 이 유형). 근본 원인은 시스템적이다: `dev-story → QA gap-fill → review` 순서에서 중간에 적는 절대 수치는 **항상 잠정값**이다. A2′가 "dev 수치 기재란 제거"를 실제 템플릿에 반영하지 않아 구조적으로 매번 발생한다. 메모리 `stale-test-count-a2`가 8회 모두 예측 적중.

2. **`(str, Enum)`의 동명-다른타입 충돌 누적(Epic 5 직렬화 부채 확대).** 3.6에서 `DeliveryStatus.HELD`(전송 인증 보류)와 2.6 `DispatchJobStatus.HELD`(구독 게이트 보류)가 **동명-다른타입**으로 공존하게 됐다(`CustomerLifecycleState.ACTIVE` vs `BaeminAuthState.ACTIVE` 선례와 동형). 현재는 타입으로 구분돼 안전하나, Epic 5 DB/API 와이어링에서 `str()`/`.value` 직렬화 시 혼동 여지가 커졌다 — A4(단일 직렬화 헬퍼)의 필요성이 증가.

3. **enum 멤버 추가가 "스토리 미명시 lock"을 깨는 패턴(메모리 적중).** 3.6의 `DeliveryStatus` 2→5멤버 변경이 스토리가 명시하지 않은 `test_idempotency.py`의 "정확히 N 멤버" lock까지 깨 함께 갱신해야 했다(메모리 `enum-member-count-locks`). "exactly N members" 잠금이 여러 파일에 흩어져 있어, enum을 늘리는 스토리는 자기 파일 외 lock을 grep으로 전수 탐색해야 한다.

4. **negative 가드 테스트의 raw-text 오탐 → AST 정밀화 반복(메모리 적중).** 3.7 send-only 가드가 초기 raw 문자열 검사로 docstring의 `getUpdates`/`TelegramUpdatePoller` 언급을 오탐 실패 → AST import-엣지 검사로 정밀화(3.1 의존성 방향 가드와 동일 교훈, 메모리 `negative-guard-tests-use-ast`). "모듈이 X를 절대 호출/import 안 함" 테스트는 처음부터 AST로 짜야 함을 8/8 스토리가 재확인.

5. **File List와 git 현실 drift(QA가 untracked 파일 추가).** 3.5에서 qa-generate-e2e가 만든 `test_idempotency_e2e.py`(untracked)가 File List에 누락돼 리뷰가 MEDIUM으로 잡았다. A2′ 수치 stale과 같은 뿌리 — dev 시점 산출물 목록이 QA 보강분을 반영 못 함. dev/QA/review 사이 산출물 동기화가 여전히 사후 보정 의존.

## 9. 핵심 takeaway

1. **"`rider_crawl` 0줄"이 실행 흐름 재배선의 안전 마진이었다.** 단방향 import + 호환 경로 보존 + 신규 의미는 전부 `rider_server` additive — 이 한 규칙이 8개 스토리, +208 테스트, 무회귀 0을 만들었다. Epic 4 `rider_agent`도 동일 규약(rider_crawl만 import, sync 유지)으로 가야 한다.
2. **"권장/격상" 으로는 시스템 문제가 안 닫힌다 — 게이트와 템플릿 변경만이 닫는다.** A1′·A3′가 3 에픽 연속, A2′가 2 에픽 연속 8/8 재발했다. 표현을 강화해도 차단 게이트가 아니면 자율 실행은 매번 더 싼 경로를 택한다. A1″는 pre-commit hook(커밋 차단), A2″는 dev 수치 입력란 물리 제거로 **행위 자체를 불가능하게** 만들어야 한다.
3. **fail-closed + 결정론 + compose-only가 코드 에픽의 3종 안전장치였다.** 모든 단계가 안전 기본값(미발송)을 갖고, 모든 비결정성을 주입으로 밀어냈으며, 기존 함수·상태머신을 재구현 없이 compose했다 — 그래서 게이트·dedup·backoff·cutover가 재현·검증 가능했다.
