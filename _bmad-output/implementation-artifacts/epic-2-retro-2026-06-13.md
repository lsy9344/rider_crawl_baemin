## 3. 잘된 점 (Successes)

1. **"소유 스토리만 손대고 인접 스토리는 미룬다" 스코프 규율의 전수 적용 — 스코프 크립 0건.** 7개 스토리가 모두 인접 작업(2.1→atomic write는 2.2, 2.3→도메인 dataclass는 2.5, 2.4→DPAPI 백엔드는 Epic 4, 게이트의 스케줄러 연동은 5.4)을 명시적으로 소유 에픽/스토리로 미뤘다. Epic 1의 핵심 무회귀 규율이 **코드 변경 맥락에서도 유지**됐다.

2. **NFR-20(기존 테스트 유지) 전 구간 무회귀.** 584→786으로 +202 케이스가 늘고 제품 코드를 실제로 고치는 동안, 기준선 통과 테스트가 새로 깨진 적이 **0건**. 특히 2.1/2.3은 "공개 동작 보존"을 골든 테스트로 잠근 채 진행했다(9탭 로딩, `ensure_ascii=False, indent=2`, `{"crawlings":[...]}` 구조, `_message_scope_key` dedup 불변).

3. **"불변식을 새 필드가 아니라 별도 레코드/뷰로 표현"하는 비침습 설계 패턴이 자리잡음.**
   - 2.3: 신규 중립 필드를 dataclass 필드가 아닌 **읽기전용 `@property`**로 만들어 `asdict()` 직렬화가 그대로 → JSON 마이그레이션 불필요.
   - 2.6: 2.5가 `==`로 잠근 `Subscription` 6필드를 깨지 않으려고, 중지 사유/시각을 도메인 필드가 아닌 별도 frozen 레코드 `SubscriptionStateChange`로 표현.
   - 2.4: 평문을 메모리에는 두되 직렬화 choke-point(`save_all`/`load_all`)에서만 `*_ref`로 추출 → 소비자(sender/2FA/UI) 무변경.

4. **fail-closed를 기본값으로 못박은 보안·운영 게이트.** 2.6 게이트는 매핑 안 된 구독 상태를 **차단**(allow=False)으로 떨어뜨리고, 성공 전송은 절대 되돌리지 않으며(`SUCCEEDED` 불변), resume가 자동 발송을 하지 않는다. 2.7 마이그레이션은 승인 없는 활성화를 조용히 건너뛰지 않고 `ValueError`로 막고, 원본 폴더를 삭제하지 않는다(NFR-18).

5. **A5의 정밀 재사용 — secret 비노출이 실측으로 지켜짐.** 2.4가 1.3 `redaction.py`와 1.1 강화 정책 위에 직접 빌드돼, 신규 설정 파일에 평문 token/password 0건. ref 포맷이 redaction 정규식과 충돌하자 `local:{id}:{field}`→`local:{id}/{field}`로 바꿔 **관측가능성(ref 추적)과 비노출을 동시에** 만족시켰다.

6. **결정론(determinism) 규율이 테스트 가능성을 끌어올림.** 2.6/2.7이 내부에서 `datetime.now()`/`uuid4()`를 호출하지 않고 호출자 주입으로 바꿔, 게이트·마이그레이션 결과가 100% 재현 가능해졌다(고정 datetime로 엣지 케이스 검증).

## 4. 어려웠던 점 / 반복 패턴 (Challenges — 무비난, 시스템 관점)

1. **테스트 수치 stale — 7/7 전수 재발 (Epic 1 A2의 직접 연장).** 모든 스토리에서 dev-story 시점 기록 수치가 이후 QA 자동화 보강(gap-fill)으로 낡았고, 리뷰가 매번 재측정해 정정했다(MEDIUM 7건 전부 이 유형). 근본 원인은 시스템적이다: `dev-story → QA 보강 → review` 순서에서 중간에 적는 절대 수치는 **항상 잠정값**이다. A2가 "미기재 또는 리뷰 정본화" 중 후자만 채택돼 지적이 구조적으로 매번 발생한다.

2. **`(str, Enum)`의 `str()` 함정(반복 관찰: 2.5 LOW, 2.7 lessons).** `json.dumps(...)`와 `==` 비교는 `"ACTIVE"`로 정상 동작하지만 `str(MigrationState.MAPPED)`는 `"MigrationState.MAPPED"`(정규화 이름)를 돌려준다. 직렬화·DB 와이어링(Epic 5)에서 `.value`를 쓰거나 `__str__`을 오버라이드하지 않으면 오작동 여지. 현재는 테스트로 잠겨 있으나 **Epic 5에 명시적 기술 부채로 이월**된다.

3. **`SecretStorageClass` 2-레이어 분기 미정합.** 2.4 `secret_store.py`는 소문자 분류(`"central"`/`"agent_local"`/`"not_stored"`)를 쓰고, 2.5 도메인은 대문자 `SecretStorageClass` enum을 새로 정의했다. 둘은 의도적으로 분리된 레이어이고 2.7이 `.upper()`로 변환해 잇지만, **단일 정본으로의 화해는 Epic 5 와이어링까지 미정**이다(현재 변환 코드 산재 위험).

4. **`crawling_index` off-by-one 위험(2.7).** 상태 폴더 출처 인덱스는 **활성 탭만 센 값이 아니라 전체 `crawlings` 배열의 1-based 위치**여야 한다. 비활성 탭이 중간에 섞이면 두 번째 활성 탭의 출처가 `crawling2`가 아니라 `crawling3`. 틀리면 dedup 해시가 빠진 빈 상태 복사가 **조용히** 일어난다. 전용 회귀 테스트로 게이트했지만, 마이그레이션류의 "조용한 손실"은 본질적으로 위험하다.

5. **선제 처리 권장 액션의 실제 미이행(A1·A3).** Epic 1이 "Epic 2 착수 전 처리 권장"한 자동 secret 게이트와 `.gitattributes`가 실제로는 처리되지 않았다. 차단이 아니었던 탓에 우선순위에서 밀렸고, Epic 2 특성상 비용이 작아 표면화하지 않았다. **코드 변경 비중이 더 커지는 Epic 3~5에서는 같은 미이행이 더 비싸진다.**

## 7. 중대 발견 / 에픽 재계획 필요 여부

**에픽 재계획 필요: 아니오.** Epic 2의 발견은 Epic 3의 계획 전제를 뒤집지 않는다. 서비스 분리·Snapshot·DeliveryLog·dry-run 방향과 도메인 모델·상태 enum 정본은 그대로 유효하다. 오히려 2.5/2.6/2.7이 `rider_server/{domain,services,migration}/` 골격을 아키텍처 정의대로 깔아 Epic 3이 그 위에 `services/`를 채우면 된다.

**전방 주의 2가지:**
1. **선제 액션 미이행 반복 위험.** A1·A3는 Epic 1·2 두 번 모두 "권장"으로 남아 미이행됐다. Epic 3은 코드/직렬화 비중이 더 커 비용이 현실화될 수 있어, A1′·A2′·A3′를 **권장이 아닌 선행 작업으로 격상**한다.
2. **도메인 모델 완성도.** 아키텍처는 13개 도메인 모델을 전제하나 2.5는 8개만 정의했다(Snapshot·Message·DeliveryLog·Job·Agent·AuthSession는 Epic 3/4/5 소유). **이는 계획된 부분 구현이지 이탈이 아니다** — Epic 3이 Snapshot·Message·DeliveryLog를 추가할 때 동일 `domain/` 규약을 따르면 된다.

## 9. 핵심 takeaway

1. **무회귀 리팩토링의 비결은 "새 필드가 아니라 새 뷰/레코드"였다.** `@property` 별칭(2.3), `SubscriptionStateChange` 별도 레코드(2.6), choke-point `*_ref` 추출(2.4)이 모두 기존 직렬화·소비자·도메인 계약을 건드리지 않고 새 의미를 더했다. Epic 3 서비스 분리도 동일 원칙(기존 `run_once` 호환 경로 보존)으로 가야 한다.
2. **"권장" 액션 아이템은 실행되지 않는다.** A1·A3가 두 에픽 연속 미이행됐고, A2는 "둘 중 하나"로 열어둔 탓에 더 싼 쪽이 선택돼 지적이 7/7 재발했다. 다음 에픽에선 **선행 작업으로 격상 + 단일 규칙 강제**가 필요하다(A1′·A2′·A3′).
3. **fail-closed와 결정론이 코드 변경 에픽의 안전 마진이었다.** 매핑 안 된 상태=차단, 승인 없음=활성화 거부(`ValueError`), `now()`/`uuid4()` 미사용으로 게이트·마이그레이션이 재현 가능·검증 가능해졌다.
