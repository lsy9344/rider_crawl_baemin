## Story

As a 운영자,
I want 3.1~3.5가 분리해 둔 수집(`CrawlService`/`SnapshotNormalizer`)·렌더(`MessageRenderService`)·전송(`DispatchFanoutService`/`IdempotentDeliveryService`) 단계의 **실패를 서로 다른 상태로 분류**하고, **인증 필요(사람 개입) 실패와 일시적(재시도 가능) 실패를 구분**해, 일시 실패는 **error_code별 제한된 backoff 재시도**(고정 5초 무한 재시도 금지)로, 인증 필요는 **무한 재시도 없이 `AUTH_REQUIRED` 보류 상태**로 남기며, **같은 Snapshot에서 일부 채널만 실패하면 실패한 채널만 재시도하고 이미 성공한 채널은 3.5 idempotency로 중복 발송되지 않게** 하고 싶다. 단, 이 모든 것은 **순수 additive·런타임 미배선**(2.5/2.6/3.1~3.5 토대 제약 계승)으로, `src/rider_crawl/`·3.1~3.5 기존 코드·`IdempotentDeliveryService.deliver_once`·`DispatchFanoutService`·기존 도메인 모델·`SubscriptionGate`/`DispatchJobStatus`(2.6)·`DeliveryLog`(3.5)는 **한 줄도 바꾸지 않고**(재사용·소비만), 신규 enum **`FailureCategory`**(NFR-15/architecture 324-325 운영 카테고리 7종 = `error_code` 어휘) + **`DeliveryStatus` 실패/재시도/보류 멤버 additive 추가**(`FAILED`·`RETRYING`·`HELD`) + 신규 서비스 **`DeliveryFailurePolicy`**(`is_retryable`/`channel_failure_category`/`backoff_delay_seconds`/`decide`=재시도 결정·`parser_warning`=반복 실패 경고·`attempt_delivery`=`deliver_once` 위에 분류+release+재시도 결정을 얹은 실패-인지 전송 primitive) + 신규 값 객체 **`RetryDecision`·`DeliveryAttemptResult`** + 신규 테스트만 추가한다,

## Acceptance Criteria

**AC1 — 수집/렌더/전송 실패를 서로 다른 상태로 분류 + 채널별 성공·실패·재시도·보류 기록 (P2-06, FR-11·26)**

1. **Given** 한 실행에서 수집은 성공하고 전송(예: Kakao)이 실패할 때 **When** 결과를 분류·기록하면 **Then** 수집 성공(`DeliveryStatus.SENT` 또는 단계별 성공 표현)과 전송 실패가 **서로 다른 상태**로 따로 보이고, 전송 실패는 운영 카테고리 `FailureCategory`(예: `KAKAO_FAILURE`)로 식별된다 — 한 단계 실패가 다른 단계/채널의 성공 상태를 덮어쓰지 않는다(단계·채널 독립). [Source: epics.md AC(631-634), architecture.md(323-325)]
2. **And** `FailureCategory`는 architecture 324-325/NFR-15의 운영 카테고리 정본과 **정확히 일치**하는 `(str, Enum)` + 멤버명==대문자값으로 **7 멤버**다: `CRAWL_FAILURE`·`AUTH_REQUIRED`·`RENDER_FAILURE`·`TELEGRAM_FAILURE`·`KAKAO_FAILURE`·`DUPLICATE_BLOCKED`·`TARGET_VALIDATION_FAILURE`(= `delivery_logs.error_code`/`jobs.error_code` 어휘). `DeliveryStatus`는 3.5의 `SENT`/`DUPLICATE_BLOCKED`에 더해 채널별 **`FAILED`(실패)·`RETRYING`(재시도)·`HELD`(보류·사람 개입)** 가 additive 추가되어 FR-26의 "성공·실패·재시도·보류"를 표현한다(총 5 멤버). [Source: architecture.md(324-325), data-api-contract.md(34·36), epics.md FR-26(66)·AC(634)]

**AC2 — 인증 필요(사람 개입) vs 일시적(재시도 가능) 실패 구분: 무한 재시도 금지 + error_code별 backoff (FR-11, NFR-4·15, ADD-15)**

3. **Given** 인증 필요 실패와 일시적 실패를 구분해야 할 때 **When** `DeliveryFailurePolicy.decide(category=..., attempt=..., max_attempts=...)`로 실패를 판정하면 **Then** `AUTH_REQUIRED`(및 `TARGET_VALIDATION_FAILURE` 같은 사람-개입 필요) 카테고리는 `RetryDecision(should_retry=False, status=HELD, delay_seconds=None)` 로 **무한 재시도하지 않고 보류 상태**로 남는다(NFR-4 "인증 필요는 무한 재시도 금지·사람 개입"). [Source: epics.md AC(636-639), architecture.md(326), data-api-contract.md(117)]
4. **And** 일시적 네트워크/서버 오류(`TELEGRAM_FAILURE`·`KAKAO_FAILURE`·`CRAWL_FAILURE`)는 **error_code별 제한된 재시도 + backoff**를 따른다 — `attempt < max_attempts`면 `RetryDecision(should_retry=True, status=RETRYING, delay_seconds=backoff_delay_seconds(attempt))`, `attempt >= max_attempts`면 `should_retry=False, status=FAILED`(재시도 소진, 운영자 가시화). `backoff_delay_seconds`는 **결정적·단조 증가·상한 있음**(예: `base·factor^(attempt-1)` capped) 으로, **고정 5초·무한 재시도를 만들지 않는다**(ADD-15 "parser 실패 backoff 없이 빠른 재시도 금지", FR-33 "error code별 backoff로 폭주 방지"). 동일 입력 → 동일 delay(내부 `now()`/`random` 미호출 — jitter는 Story 5.4 주입). [Source: epics.md AC(636-639), architecture.md(327), data-api-contract.md(36 `attempts`/`run_after`)]

**AC3 — parser 반복 실패 → 운영자 확인 경고 상태 (FR-11)**

5. **Given** 같은 parser 작업이 계속 실패할 때 **When** `DeliveryFailurePolicy.parser_warning(consecutive_failures, threshold=...)`로 판정하면 **Then** 연속 실패가 threshold 이상이면 **운영자가 확인할 수 있는 경고(True)** 로 전환되고, threshold 미만이면 False다(반복 실패 parser를 조용히 빠르게 재시도하지 않고 가시화 — ADD-15·FR-33). 본 스토리는 **경고 판정(boolean/상태 전환 결정)** 까지이며, platform-wide circuit breaker는 Story 5.4 소유다. [Source: epics.md AC(641-643), architecture.md(327-328), ops-security-test-contract.md(29·95)]

**AC4 — 같은 Snapshot 일부 채널 실패: 실패 채널만 재시도, 성공 채널 중복 발송 없음 (FR-26, 3.5 idempotency 재사용)**

### insert-then-send와 release의 관계 (AC4 — 3.5가 남긴 명시 위임, 놓치기 쉬움)

- **3.5가 남긴 gap:** `deliver_once`는 reserve(key)→send(job) 후 send가 예외를 던지면 **분류·release 없이 전파**하고 reserve된 key를 **회수하지 않는다**(3.5 의도 — "release = 3.6"). 그래서 3.5 단독에서는 일시 실패 후 재시도해도 같은 key의 reserve가 충돌→`DUPLICATE_BLOCKED`→**재전송 안 됨**(fail-closed, 3.5 e2e 테스트 (B)가 이를 단언). 본 스토리는 **재시도 가능한 일시 실패에 한해** 그 미발송 key를 `release(key)`로 풀어 재시도 길을 연다 — 이게 "실패한 채널만 재시도"(FR-26 AC4)의 핵심 메커니즘이다.
- **release는 재시도 가능 실패에만.** `attempt_delivery`는 `decision.should_retry`가 True일 때만 `release(key)`를 호출한다 — `AUTH_REQUIRED`/`TARGET_VALIDATION_FAILURE`(`HELD`)·`RENDER_FAILURE`(`FAILED`)·소진(`FAILED`)은 release하지 **않는다**(보류·종료 상태 유지, fail-closed: 오발송보다 미발송). 성공(`SENT`)·중복(`DUPLICATE_BLOCKED`)은 `deliver_once`가 처리하므로 애초에 release 경로로 오지 않는다.
- **성공 채널 중복 차단은 3.5 그대로.** 일부 채널만 실패해 재시도할 때, 이미 `SENT`로 key를 확보한 성공 채널은 2라운드에서 reserve 충돌→`DUPLICATE_BLOCKED`(send 미호출)다 — 3.5 idempotency가 그대로 "이미 성공한 채널 중복 발송 없음"(FR-26·NFR-3)을 보장한다. 본 스토리는 그 위에 "실패 채널 key 회수"만 더한다. [Source: src/rider_server/services/idempotency.py(96-107·120-133), src/rider_server/services/test 참조 test_idempotency_e2e (B), architecture.md(313-314·346)]
- **실제 release 실행은 Epic 5.** `release` 콜백 자체(미발송 `delivery_logs`/`jobs` reservation 행 삭제 또는 status 전이)와 재큐잉(`jobs.attempts++`·`run_after` 갱신)은 Epic 5 영속·async wiring이 한다. 본 스토리는 **결정(release 할지 말지)** 과 **in-memory release seam**(테스트)만 제공한다(2.5/2.6/3.1~3.5 "정의만, 런타임 미배선"). [Source: architecture.md(308-315·428), data-api-contract.md(36)]
