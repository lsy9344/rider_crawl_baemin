## Story

As a 운영자,
I want 고객 **구독 상태(`SubscriptionStatus`)** 가 작업 실행 가능 상태인지 판정하는 **순수 `SubscriptionGate` 정책 로직**을 만들어, 중지(`SUSPENDED`)·해지(`CANCELLED`) 고객의 **신규 CrawlJob/DispatchJob 생성을 막고**, 중지 시 미전송 Dispatch는 **`HELD`** 로 보류하되 **이미 성공 기록된 전송은 절대 재발송하지 않으며**, 복구 시 보류분은 **운영자 확인 후에만** 폐기/재개되게 하고 싶다,
so that 결제 실패·수동 중지 고객에게 실수로 실적이 계속 수집·발송되지 않고(fail-closed), 이후 scheduler(Story 5.4)·Admin 상태 전이 UI(Story 5.7)·DB/audit(Epic 5)가 **동일한 게이트 정책 정본**을 재사용한다(FR-6, ADD-9 Subscription execution gate).

> **이 스토리의 성격 — "순수 게이트 정책(pure policy/service) 로직만." scheduler 연동도, DB/영속도, Admin UI도, 전체 Job 모델 정의도 아니다.** Epic 2는 "도메인 모델·상태(enum) 진화" 에픽이고(epics.md 353-355), 본 스토리의 FR-6 deliverable은 **"`SubscriptionGate`가 `SubscriptionStatus`(2.5에서 정의됨)를 입력으로 받아 ① 신규 작업 예약/전송 허용 여부, ② 중지 시 미전송 Dispatch의 `HELD` 보류, ③ 복구 시 `HELD`의 운영자-결정 기반 폐기/재개를 **결정론적 순수 함수**로 판정하고, 그 정책이 테스트로 잠긴다"** 이다. **scheduler 앞단 wiring은 Story 5.4(FR-6, P4-04), Admin 상태 전이 UI는 Story 5.7, PostgreSQL/Alembic·Pydantic 스키마·`jobs` 테이블 영속은 Epic 5(P4-02~04), 전체 Job lifecycle(CLAIMED/RUNNING/FAILED/RETRY 등)·DispatchService 실행은 Epic 3/4/5 소유다.** 본 스토리는 그 게이트가 쓸 **정책 함수와 최소 의사결정 어휘만** 둔다. [Source: epics.md Story 2.6(463-485)·FR Coverage Map FR-6(159), epics.md Story 5.4(980-1001)·5.7(1051-1073), architecture.md(425-426·485-488), data-api-contract.md Subscription execution gate(112-119)]

## Acceptance Criteria

**AC1 — `SubscriptionGate`가 구독 상태로 신규 작업 예약/전송 허용 여부를 판정 (FR-6, ADD-9, data-api-contract Subscription execution gate)**

1. **Given** 고객 구독 상태가 실행 게이트로 쓰일 때 **When** `SubscriptionGate`가 `Subscription`(2.5)의 `status: SubscriptionStatus`를 평가하면(FR-6) **Then** **`PAYMENT_ACTIVE`/`PAYMENT_FAILED_GRACE`가 아닌**(즉 `SUSPENDED`/`CANCELLED`) 고객은 **신규 CrawlJob도 신규 DispatchJob도 생성/예약되지 않도록**(`allow_new_crawl_job=False`, `allow_new_dispatch_job=False`) 판정되고, 판정 결과는 frozen `GateDecision`(허용 플래그 + `warn_admin` + 기계가독 `reason`)로 반환된다. [Source: epics.md AC1(471-475), data-api-contract.md(112-119), architecture.md(485-488)]
2. **And** **`PAYMENT_FAILED_GRACE`** 고객은 수집/전송을 **계속 허용**(`allow_new_crawl_job=True`, `allow_new_dispatch_job=True`)하되 **Admin 경고를 표시할 수 있도록**(`warn_admin=True`) 판정된다(결제 실패 유예 — "continue collection/delivery but show Admin warning"). [Source: epics.md AC1(474), data-api-contract.md(117)]
3. **And** **`SUSPENDED`** 는 신규 CrawlJob/DispatchJob 생성을 **멈추되**, 게이트는 **설정·secret/profile 참조를 절대 건드리지 않는다**(비파괴) — 게이트는 작업 생성만 막고 `PlatformAccount`/`SecretRef`/`BrowserProfile`을 삭제·변경하지 않는다(보존). `CANCELLED`도 신규 작업 생성을 막는다(secret revoke/profile archival 같은 retention 처리는 Epic 5 소유 — 게이트는 차단 + `reason="CANCELLED"`만 판정). [Source: epics.md AC1(475), data-api-contract.md(118-119)]

**AC2 — 중지 전환 시 미전송 Dispatch는 `HELD`, 성공 기록은 재발송 금지 (FR-6)**

4. **Given** 중지 고객의 미전송 작업이 있을 때 **When** 고객이 `SUSPENDED`로 전환되고 게이트가 각 Dispatch의 보류를 판정하면 **Then** **미전송(`PENDING`) Dispatch는 `HELD`로 전환**되어 자동 발송되지 않는다(`hold_undelivered(PENDING) == HELD`). 이미 `HELD`인 것은 `HELD` 유지(멱등). [Source: epics.md AC2(477-479), epics.md FR-6(31)]
5. **And** **이미 성공 기록된(`SUCCEEDED`) Dispatch는 구독 상태 변경 후에도 재전송되지 않는다** — 게이트는 `SUCCEEDED`를 `HELD`로도, 발송 가능 상태로도 되돌리지 않는다(`hold_undelivered(SUCCEEDED) == SUCCEEDED`, 어떤 게이트 함수도 `SUCCEEDED`를 재발송 가능 상태로 전이하지 않음 — fail-closed 불변식 ①). [Source: epics.md AC2(480), data-api-contract.md(118), architecture.md(329)]

**AC3 — 복구 시 `HELD`는 운영자 확인 후에만 처리, 중지 사유·시각 기록 (FR-6, FR-30)**

6. **Given** 중지에서 복구할 때 **When** 고객이 `SUSPENDED`에서 `ACTIVE`/`PAYMENT_ACTIVE`로 복구되면 **Then** `HELD` Dispatch는 **복구 사실만으로 자동 발송되지 않고**(불변식 ②), **운영자의 `HeldDisposition` 입력이 있어야** 폐기 또는 재개 중 하나로 처리된다: `dispose_held(HELD, DISCARD) == DISCARDED`, `dispose_held(HELD, RESUME) == PENDING`(재발송 후보로 복귀). `HELD`가 아닌 상태(특히 `SUCCEEDED`)에 `dispose_held`를 호출하면 **재발송/오처리를 막기 위해 명확한 예외**(`ValueError`)를 낸다(fail-closed). [Source: epics.md AC3(482-484), epics.md FR-30(73·130), architecture.md(329)]
7. **And** 중지/복구 전이 시 **중지 사유와 마지막 상태 변경 시각이 상태 전이 기록(`SubscriptionStateChange`)에 남는다**: `SubscriptionGate.suspend(sub, reason=..., at=...)`·`resume(...)`는 `(상태가 바뀐 새 Subscription, SubscriptionStateChange(subscription_id, from_status, to_status, reason, changed_at))` 를 반환하고(frozen·불변, `dataclasses.replace`로 status만 전이해 나머지 식별·이력 필드 보존), 이 기록이 Epic 5에서 `audit_logs`로 영속될 정본 데이터다(FR-30 "중지 사유·시각 표시"). [Source: epics.md AC3(485), epics.md FR-30(73), data-api-contract.md(38·119), architecture.md(317-319)]

### `HELD` 어휘와 Dispatch 상태 (AC2/3 — 최소 부분집합, Epic 5 reconcile)

- **`HELD`는 contract `jobs` 테이블 status에 열거돼 있지 않다** — FR-6(epics 31·479)·readiness(92·130)가 정책으로만 명시한다. 본 스토리는 게이트가 **보류/재발송 금지를 판정하는 데 필요한 최소 어휘**로 `DispatchJobStatus(PENDING/HELD/SUCCEEDED/DISCARDED)`를 게이트 모듈에 정의한다. **전체 `jobs` 상태 정본(CLAIMED/RUNNING/FAILED/RETRY 등)·dedup·DeliveryLog "성공 기록"은 Epic 3/4/5 소유**이고, 그때 본 부분집합과 reconcile한다(2.5가 enum을 소비 모델보다 먼저 정의한 선례와 동형). docstring에 "gate-facing subset" 명시. [Source: data-api-contract.md(36·146-156), epics.md FR-6(31), 2-5 스토리(24-25)]
- **"이미 성공 기록된 Dispatch"** = Epic 3의 `DeliveryLog` 성공 행을 가진 dispatch. 본 스토리에선 `DispatchJobStatus.SUCCEEDED`로 추상화하고, 게이트가 그것을 **절대 재발송 가능 상태로 되돌리지 않음**을 보장·테스트한다(불변식 ①). 실제 dedup 유니크 제약(`uq_delivery_logs_dedup_key`)은 Epic 3/5. [Source: epics.md AC2(480), architecture.md(174·357), data-api-contract.md(146-156)]

### fail-closed 불변식 (NFR — 오발송보다 미발송, 깨면 안전 위반)

1. **성공분 재발송 0:** `SUCCEEDED` Dispatch는 `hold_undelivered`로도 `dispose_held`로도 발송 가능/대기 상태로 전이되지 않는다. `dispose_held(SUCCEEDED, *)`는 `ValueError`. [Source: epics.md AC2(480), architecture.md(329)]
2. **복구 ≠ 자동 발송:** `resume()`는 구독 상태만 바꾸고 `HELD` Dispatch를 건드리지 않는다. `HELD`→발송 가능은 **오직** 운영자 `HeldDisposition.RESUME` 입력 시. [Source: epics.md AC3(482-484), epics.md FR-30(130)]
3. **미지 상태 차단:** `evaluate`의 미매핑/예상 외 상태는 허용이 아니라 차단(또는 `ValueError`). [Source: architecture.md(329·556), project-context.md(86-89)]

### "중지 사유·시각"을 도메인 필드 추가 대신 `SubscriptionStateChange`로 (AC3 핵심 설계)

- AC3·FR-30이 요구하는 "중지 사유와 마지막 상태 변경 시각 기록"을 **`Subscription` dataclass 필드 추가로 구현하지 않는다.** 이유: 2.5의 `test_domain_models.py::test_model_field_sets_match_contract`(86-93)가 `Subscription`의 **정확한 6개 필드 집합**을 `==`로 잠갔고, 필드를 추가하면 이 회귀 테스트가 깨진다(NFR-20 위반 — Epic 2 전체가 "기존 무변경/회귀 0"을 최우선으로 규율). 또한 contract `subscriptions` 행(26)에도 `suspended_reason`/`changed_at`이 없다 — 상태 변경 이력의 정본은 `audit_logs`(Epic 5)다. [Source: tests/server/test_domain_models.py(84-93), data-api-contract.md(26·38), epics.md Epic 2(353-355)]
- 대신 게이트의 `suspend()`/`resume()`가 **`SubscriptionStateChange`(불변 값 객체)** 를 반환해 사유·시각·from/to를 담는다. 이것이 FR-30 "중지 사유·시각"의 정본 데이터이고 Epic 5가 `audit_logs`로 영속한다. 이렇게 하면 도메인 무변경(회귀 0) + AC3 충족을 동시에 만족한다. [Source: epics.md AC3(485), data-api-contract.md(38), architecture.md(317-319)]
