## Acceptance Criteria

**AC1 — 실발송 없는 dry-run 실행: 수집·렌더만, 대표 배민/쿠팡 성공 (FR-3, operations-security-test E2E dry-run)**

1. **Given** 신규 분리 경로(`CrawlService`(3.1)·`MessageRenderService`(3.3))가 있을 때 **When** `run_dry_run(config, *, crawl_snapshot=<주입>, target_id, message_id, snapshot_id, source_label="", now=None)`를 실행하면 **Then** `CrawlService.crawl(config, crawl_snapshot=crawl_snapshot)` → `MessageRenderService.render_message(snapshot, message_id=, snapshot_id=, source_label=, now=)`만 compose해 신규 `Message`를 만들고 `DryRunResult(target_id, message, sent=False)`를 반환한다 — **실제 Telegram/Kakao 발송은 일어나지 않는다**(`run_dry_run` 시그니처에 sender/`send` 콜백이 0이고, `DispatchService`/`DispatchFanoutService`/`CentralTelegramSender`를 호출하지 않으며, `DryRunResult.sent`는 **항상 False**). [Source: epics.md AC1(679-682), operations-security-test-contract.md(48), src/rider_server/services/crawl_service.py(34-42), src/rider_server/services/message_render_service.py(58-91)]
2. **And** 대표 **배민 대상**(`CurrentScreenSnapshot` fake)과 **쿠팡 대상**(`PerformanceSnapshot` fake) 둘 다에 대해 `run_dry_run`이 성공한다(외부 브라우저/Telegram 미호출 — fake `crawl_snapshot` 주입, fail-closed: 필수 데이터 누락 시 `render_current_screen_message`/`MissingPerformanceDataError`가 raise해 **잘못된 메시지를 만들지 않는다**). [Source: operations-security-test-contract.md(48), project-context.md(36·55), src/rider_server/services/snapshot_normalizer.py(38-43)]

**AC2 — 기준선 비교 + 차이 시 자동 활성화 차단(승인 게이트) (FR-3, ADD-16 6~7단계, NFR-22)**

3. **Given** 기존/신규 메시지 차이를 검토해야 할 때 **When** `compare_to_baseline(result, *, seed=<MigrationSeed | None>)`로 dry-run 결과를 기준선과 비교하면 **Then** 2.7이 승계한 `MigrationSeed.message_hash`(old `last_message` hash)와 신규 `result.message.text_hash`를 **hash 대 hash로 비교**해 `DryRunComparison(target_id, matches, baseline_hash, new_hash, preview_redacted)`를 반환한다 — `matches=(seed is not None and seed.message_hash == result.message.text_hash)`, `seed=None`(첫 발송 전)이면 `matches=False`(차이로 취급, fail-closed), `preview_redacted`는 `redact()` 통과·길이 cap. [Source: epics.md AC2(684-687), src/rider_server/migration/runner.py(86-98·332-353), src/rider_server/domain/message.py(28), src/rider_server/services/message_render_service.py(81-82)]
4. **And** **메시지 차이가 발생하면 신규 DeliveryRule 활성화가 자동으로 진행되지 않는다** — `approve_after_review(target, comparison, *, operator_acknowledged_diff)`는 `comparison.matches`가 False인데 `operator_acknowledged_diff`가 False면 **`CutoverApprovalError`를 raise**(fail-closed: 운영자가 차이를 확인·승인하지 않은 대상은 `APPROVED`로 전이 불가)하고, `matches=True`이거나 운영자가 차이를 명시 승인(`operator_acknowledged_diff=True`)한 경우에만 2.7 `approve(target)`(=`DRY_RUN_PASSED`→`APPROVED`)에 compose한다. 즉 **운영자가 차이를 확인·승인한 대상만** 이후 `activate`로 신규 DeliveryRule 전송 대상이 된다. [Source: epics.md AC2(686-687), implementation-contract.md(104-105), src/rider_server/migration/runner.py(`approve` 424-426·`activate` 429-433), 2-7 스토리(50·85)]

### 위치 결정 — 왜 `migration/cutover.py`인가 (반드시 읽을 것)

- **architecture가 `rider_server/migration/`을 cutover 정본 위치로 못 박는다**(442: "cutover 상태머신, kill switch, canary"·507: "⑧ 마이그레이션/배포/안전(FR-27~34) → rider_server/migration/"·153-154: "마이그레이션 cutover: 상태머신(discovered→…→rolled back) + 전역 kill switch + old/new 동시전송 방지 + canary"). dry-run 비교·old/new 동시전송 방지·rollback은 **마이그레이션 cutover 개념**이라 `services/`(파이프라인 서비스)가 아니라 `migration/`에 둔다.
- **2.7이 이 디렉터리를 신설하며 본 스토리에 명시 위임했다.** `migration/runner.py`의 `mark_dry_run_passed`(419-421)는 "실제 old/new 렌더 비교는 Story 3.8", `activate`(429-433)는 "실제 DeliveryRule enable/발송은 Story 3.4 + Epic 5", 2.7 스토리 cutover 표(124-125)는 "6. dry-run 1회 실행 → Story 3.8 / 7. old/new 비교 → Story 3.8". 비교 기준선(`MigrationSeed`)도 `migration/` 소속이다. 따라서 본 스토리는 `migration/cutover.py`로 두어 **2.7 상태머신·seed와 한 패키지**에서 compose한다. Epic 5/P6가 같은 디렉터리에 운영 cutover(kill switch·canary·DB 영속) wiring을 additive로 덧붙인다. [Source: src/rider_server/migration/runner.py(419-433·441-443), 2-7 스토리(124-125·56·110-111), architecture.md(442·507·153-154)]
- **`migration → services` import는 정상**이다. 마이그레이션은 **오케스트레이션 레이어**라 하위 정책(`services/`)·도메인(`domain/`)을 import해 compose한다(2.7 `runner.py`가 이미 `rider_crawl` + `domain`을 orchestrate). 의존성 절대 규칙은 `rider_crawl ↛ rider_server`(역방향 금지)뿐이고, `rider_server` 내부 레이어링(domain ← services ← migration)은 본 import 방향과 일치한다(`test_rider_crawl_never_imports_rider_server`는 영향 없음 — 본 코드는 `rider_server` 내부 import). [Source: project-context.md(64), tests/server/test_run_once_split.py(431-448), src/rider_server/migration/runner.py(38-50)]
- **`DryRunResult`/`DryRunComparison`/`CutoverResult`/`RollbackResult`는 migration-레이어 값 객체**다 — `TargetMigration`/`MigrationSeed`/`MigrationResult`(2.7)가 domain이 아니라 migration 값 객체였던 선례와 동형. 그래서 `domain/__init__.py`·model-count lock 무변경(3.7과 동일). [Source: src/rider_server/migration/runner.py(77-128)]

### "dry-run = 구조적 no-send" 정합 (AC1 — 가장 헷갈리는 지점)

- **dry-run은 수집·렌더·저장까지만, 전송 0.** `run_dry_run`은 `CrawlService.crawl`(3.1) → `MessageRenderService.render_message`(3.3)만 compose한다. **`send`/sender 콜백을 인자로 받지 않고**, `DispatchService`(3.1)·`DispatchFanoutService.dispatch_all`(3.4)·`CentralTelegramSender.send`(3.7)를 import·호출하지 않는다 → 실발송 경로가 코드에 **존재하지 않는다**(구조적 보장, "send_enabled=False" 같은 런타임 플래그에 의존하지 않음). `DryRunResult.sent`는 항상 False. 이것이 operations-security-test-contract 48행 "Collection/render/store succeeds with **actual sending disabled**"의 정확한 구현이다.
- **"저장(store)"의 의미.** AC1의 "저장"은 신규 `Message`(렌더 산출물·비교 정본)를 만드는 것까지다. 실제 DB 영속(`messages`/`snapshots` 테이블)은 Epic 5. 필요 시 `SnapshotNormalizer.normalize`(3.2)로 정규화 `Snapshot`을 만들 수 있으나(fail-closed 저장), 본 스토리 핵심은 비교 대상인 `Message` 생성이고 렌더러가 이미 `MissingPerformanceDataError`로 fail-closed라 정규화는 선택적이다(과한 wiring 금지 — 호출부/Epic 5가 정한다). [Source: operations-security-test-contract.md(48), src/rider_server/services/snapshot_normalizer.py(38-43·52-60), project-context.md(36)]
- **fail-closed(오발송보다 미발송).** crawl/render 예외는 삼키지 않고 전파한다(3.1 `crawl_service.py` 17-19·3.2 fail-closed). 누락 데이터로 0/기본값 메시지를 만들지 않아 잘못된 dry-run 비교를 막는다. [Source: src/rider_server/services/crawl_service.py(17-19·41), project-context.md(36·87)]

### 기준선 비교와 2.7 `MigrationSeed`의 관계 (AC2)

- **기준선 = old `last_message` hash(`MigrationSeed.message_hash`).** 2.7 `seed_from_state`(332-353)가 old `runtime/state/crawling{N}/last_message.<scope>.sha256` 파일 내용을 `MigrationSeed.message_hash`로 승계했다(NFR-21, ADD-16 5단계). 본 스토리는 그 hash를 **정본 기준선**으로 쓰고 신규 `Message.text_hash`(3.3 = `sha256(text)`)와 **hash 대 hash** 비교한다 — old 평문 메시지는 보관되지 않으므로(hash만 승계) 재수집·재렌더 비교가 아니라 hash 동일성 비교다. 같은 메시지면 같은 hash(3.1 `message_hash`·3.3 `text_hash`·old `last_message` 모두 `sha256(text)` 동일 계산식). [Source: src/rider_server/migration/runner.py(86-98·332-353), src/rider_server/services/message_render_service.py(79-80), src/rider_server/services/dispatch_service.py(21-22·63), implementation-contract.md(102·104)]
- **seed 없음 = 차이로 취급(fail-closed).** 첫 발송 전 탭은 `MigrationSeed`가 없다(`seed_from_state`가 `None`). 이때 `matches=False`로 둬 자동 활성화를 막는다(운영자가 신규 메시지를 직접 확인·승인해야 함 — 오발송보다 미발송). [Source: src/rider_server/migration/runner.py(340-341), project-context.md(87)]
- **차이 시 자동 활성화 차단 = 승인 게이트.** AC2의 핵심은 "차이가 있으면 활성화가 자동 진행되지 않고, 운영자가 확인·승인한 대상만 전송"이다. 2.7 상태머신이 이미 `MAPPED`→`DRY_RUN_PASSED`→`APPROVED`→`ACTIVE`로 **승인(`approve`)을 명시 전이로** 강제한다. 본 스토리 `approve_after_review`는 그 위에 **"차이가 있는데 운영자가 확인 안 함 → `CutoverApprovalError`"** 한 겹을 더해 *코드로* 차단한다(2.7 `approve` 자체는 차이 여부를 모름 — 본 함수가 비교 결과를 게이트에 주입). [Source: src/rider_server/migration/runner.py(407-433), 2-7 스토리(50·85), implementation-contract.md(105)]

### cutover 동시전송 방지(NFR-24)와 rollback dedup 보존(NFR-25) (AC3 — 놓치기 쉬운 disaster 방지)

- **NFR-24 위험:** old 런타임 경로(legacy `run_once`/UI 폴러)와 new `DeliveryRule`(중앙 경로)이 같은 대상에 대해 **동시에 실제 전송**하면 고객이 같은 실적을 두 번 받는다(중복 발송 disaster). 안전 규칙: cutover 시 **둘 중 하나는 반드시 꺼져 있어야 한다.** `assert_no_dual_active_send`는 `legacy_path_active and new_rule_enabled` → `DualSendError`(fail-closed)로 이를 코드로 강제하고, `activate_cutover`는 legacy가 켜진 채 신규 enable을 시도하면 거부한다(legacy를 먼저 끄고 활성화). 본 스토리는 "전송 가능 상태"를 불리언으로 받는 **순수 가드**이고, 실제 legacy 프로세스 종료·런타임 스위치는 Epic 5다. [Source: epics.md NFR-24(115), architecture.md(77·153-154), project-context.md(92·94)]
- **NFR-25 rollback 3원칙:** (a) **신규 DeliveryRule 비활성화** = `replace(rule, enabled=False)` soft-delete(물리 삭제 금지, 2.5 `DeliveryRule.enabled` 패턴 — `delivery_rule.py` 5행 "비활성화는 물리 삭제가 아니라 enabled=False"), (b) **기존 런타임 경로 복구** = `legacy_path_restored=True` 표현(실제 복구 스위치는 Epic 5; 2.7 백업 `backup_path`가 복구 아티팩트), (c) **신규 로그는 중복 방지 기록으로 보존** = 입력 `delivery_logs`를 **변경·삭제 없이** `preserved_logs`로 반환. 이게 핵심: rollback 후 legacy가 다시 보내도 신규 `DeliveryLog`의 dedup 기록이 남아 있어(3.5 dedup key) **같은 메시지가 중복 전송되지 않는다.** 로그를 지우면 dedup seed가 사라져 rollback이 중복 발송을 유발하는 disaster가 된다 — 그래서 **보존이 fail-closed 정답**이다. [Source: epics.md NFR-25(116), src/rider_server/domain/delivery_rule.py(5·20), src/rider_server/domain/delivery_log.py(29-37), src/rider_server/services/idempotency.py(83-142), 2-7 스토리(134-144 백업)]
