## 1. 에픽 요약과 지표

Epic 5는 **MVP(P0~P4)의 마지막 에픽**이자, Epic 2~3가 "런타임 미배선 순수 도메인/서비스"로, Epic 4가 "outbound-only Agent 클라이언트(서버 stub/mock 대상 검증)"로 남겨둔 자리를 **실체로 채우는 에픽**이다. 신규 의미를 전부 `src/rider_server/`에 가산하되, Epic 4가 sync 런타임이었다면 Epic 5의 `rider_server`는 **FastAPI/SQLAlchemy 2.x(async) + Alembic + PostgreSQL** 비동기 런타임이다 — 즉 패키지 격리 규약(단방향 import)은 그대로 두되 **async가 정본**인 첫 에픽이다.

11개 스토리가 ① FastAPI 스캐폴딩·헬스/버전/메트릭(5.1) ② 14테이블 ORM·Alembic(5.2) ③ `QueueBackend` 추상화 + `FOR UPDATE SKIP LOCKED` job queue(5.3) ④ jitter·circuit breaker·구독 게이트 scheduler(5.4) ⑤ Telegram webhook `/register`·채널 검증·활성화(5.5) ⑥ Admin 대시보드·상태 심각도(5.6) ⑦ 수동 운영 액션·구독 상태 전이(5.7) ⑧ audit log·MFA·4역할·token revoke·backup/restore(5.8) ⑨ 7개 모니터링 지표·알림·runbook(5.9) ⑩ 100 fake target 부하 smoke·negative safety(5.10) ⑪ Admin 엔티티 CRUD UI(5.11)를 다뤘다.

핵심 성격: Epic 5는 **"관측성·제어·보안·복구성을 갖춘 운영 통제면(control plane)"**을 완성했다. 단, **자율 수집→렌더→전송 런타임은 의도적으로 미배선**이다(아래 §4-5, §7 — 현재 발송 chokepoint는 운영자 주도 `test_send` 하나, 중앙 dispatch 루프·`crawl_worker.py`는 미존재). MVP 통제면은 완성, 자동 운영 런타임은 cutover/후속 워커의 몫으로 명확히 경계가 그어졌다.

**연속성 핵심(2가지 명제의 다섯 번째 증거):**
1. **"코드/도구로 닫은 것은 닫힌다"** — A7-carry(server async 가드)가 5.1 코드 한 번으로 11/11 자동 작동했다. Epic 4의 4.1 AST 가드와 동일 메커니즘이 async 측에서 재현돼, "write-once, inherit" 가드 전략이 sync·async 양쪽에서 입증됐다.
2. **"사람 의지에 맡긴 것은 안 닫힌다"** — A1·A2·A3는 Epic 4가 "차단 게이트로 격상"이라는 최강 표현을 썼음에도 **다섯 번째로 미이행**. 자율 실행은 매번 더 싼 경로(per-story redact, dev 시점 잠정 수치, `\n` 관행)를 택했다. A1은 가장 비싼 에픽(secret 표면 폭증)에서도 부재, A2는 5.8 단 1건만 예외, A3는 비용이 잠복했을 뿐. **다섯 에픽 연속 동일 결론** — 이 셋은 더 이상 회고 권장의 문제가 아니라, 다음 작업 단위 진입의 **전제 게이트**로 자동화돼야만 닫힌다.

1. **server-only 에픽 — `rider_crawl`·`rider_agent` 동시 0줄, 3-패키지 위계 완전 보존.** 11개 스토리가 `src/rider_crawl/`·`src/rider_agent/`를 단 한 줄도 고치지 않고(`git diff -w 3d4cac0..HEAD` 양쪽 빈 출력) `src/rider_server/`에만 59 files/+9664를 가산했다. Epic 2~3가 만든 도메인 11모델·서비스 7모듈을 **0줄 재사용**(5.4 `SubscriptionGate`·5.3 queue·5.7 `IdempotentDeliveryService`·`DispatchService` compose). Epic 3의 "0줄=무회귀 안전마진"이 다섯 번째 에픽까지 계승됐다.

2. **A7-carry 즉시 이행 = server async 경계 가드를 5.1에 선제 배치.** Epic 4 회고의 "server는 async 정본이니 sync 가드를 그대로 쓰면 안 된다"를 **착수 첫 스토리에서 코드로** 닫았다. `test_server_async_boundary.py`가 blocking sync를 async에서 직접 호출하는 모든 회귀를 11/11 자동 차단했고, 5.5는 동기 `urllib` 전송을 `run_in_executor`로 경계 밖에 두는 패턴으로 가드를 통과했다 — "회고 액션이 다음 에픽 첫 스토리의 코드가 된" 모범 사례.

3. **14테이블 lock·9-dep lock·enum count-lock 전 구간 무파손 — "추가는 가산, 정본은 불변".** PostgreSQL 영속·queue·scheduler·security·audit를 전부 올리면서도 **신규 테이블 0**(모든 신규 의미는 additive 컬럼 또는 plain-string 상수), 메인 deps 9개 고정(FastAPI/SQLAlchemy/asyncpg/jinja2는 전부 optional group), 기존 enum 멤버 수정 0. 신규 enum 3종(`MessengerChannelState`/`AdminRole`/`AuditResult`)도 각자 count-lock을 달고 들어왔다. 메모리 `db-tables-13-vs-14`·`server-deps-go-in-optional-group`·`enum-member-count-locks`가 그대로 코딩 규약으로 작동.

4. **"재구현 금지, compose만" 규율이 정책 중복·임계값 drift를 원천 차단.** 5.4 scheduler는 jitter·breaker·구독게이트·backoff를 **새로 짜지 않고** `SubscriptionGate`·`DeliveryFailurePolicy`·`QueueBackend`를 조립만 했고, 5.9 metrics는 임계값을 scheduler·severity **원본과 identity/equality로 잠가**(`AGENT_OFFLINE_AFTER is severity.AGENT_OFFLINE_AFTER`) 다중 소스 drift를 차단했다. 5.10은 negative safety를 **재구현 대신 AST 추적 매트릭스**(7 시나리오를 기존 통과 테스트로 cross-ref + 허위완료 검출기)로 입증.

5. **fail-closed가 server 전 경로 관통 — 기본값이 곧 안전.** 5.3 lease 소유권 미일치 시 double-success 차단(409/410) → 5.4 구독 비ACTIVE는 신규 job 예약 0 → 5.5 검증 전 채널은 운영 전송 0(`/register` 후 VERIFIED→ACTIVE만) → 5.6 인증필요/CENTER_MISMATCH는 시간경과 심각도보다 우선 표시 → 5.7 SUCCEEDED는 발송가능 회귀 불가·retry는 dedup 우회 0 → 5.8 **모든 보안 seam 기본 deny**(principal None→401·MFA 미검증→403·역할 부족→403·IP 불허→403·non-sending 기본 ON) → 5.10 kill switch 기본 OFF(`RIDER_SENDING_ENABLED` 미설정=차단) → 5.11 평문 secret→400. 안전 기본값이 한 스토리도 빠짐없이 적용.

6. **레이어 물리 분리가 가드 충돌을 구조적으로 해소 + 보안 삽입점을 예비.** 5.6이 admin 모듈을 "읽기 전용"으로 AST 가드(write-call 0)하자 5.7의 쓰기 액션과 충돌할 뻔했으나, **액션을 별 모듈(`actions_routes.py`)로 분리**하고 read-only 가드 scope를 화이트리스트로 좁혀 해결. 그 분리 덕에 5.8 보안 게이트(역할·MFA·audit-on-deny)가 **기존 stub seam을 fail-closed 강제기로 교체**하는 형태로 깔끔히 삽입됐다 — 5.6 fail-safe 스텁 → 5.8 fail-closed 강제기. "5.6=보기, 5.7=하기, 5.8=지키기"의 소유 경계가 일관 유지.

7. **메모리 자산이 또 적중 — 같은 함정 사전 회피.** `jobs-complete-route-wallclock-now`(5.3 lease 만료 테스트 실 now)·`pg-gated-files-hide-pure-helpers`(5.2·5.3·5.4 always-run 순수 테스트 분리)·`admin-routes-wallclock-severity`(5.6 심각도는 순수/서비스 층에서 검증, 라우트는 STOPPED/auth/empty만)·`story-5-8-audit-on-deny-anti-flooding`(5.8 인증 주체 거부만 DENIED audit, None→401은 audit 없음)·`redact-skips-operational-ids`(5.9 미인증 fleet scrape에 식별자 미탑재)가 모두 실 사용. Epic 4 메모리 8종 + Epic 5 신규 5종이 누적 자산으로 작동.

## 4. 어려웠던 점 / 반복 패턴 (Challenges — 무비난, 시스템 관점)

1. **테스트 수치 stale — 거의 전수 재발(Epic 1→2→3→4→5, 다섯 에픽 연속).** `dev-story → qa-generate-e2e gap-fill → review` 순서에서 dev 시점 절대 수치는 **구조적으로 항상 잠정값**이다. Epic 5도 9/11 스토리가 동일 MEDIUM(5.8만 예외)으로 재측정·정정됐고, File List drift도 2건(5.4·5.6). 보정은 11/11 작동했고 회귀 0이지만, A2‴가 "dev 수치 입력란 물리 제거 + qa-e2e 자동 기입"을 **실제 템플릿에 반영하지 않아** 매번 발생한다. **다섯 번째 연속은 "도구로 닫아야 할 문제"임을 재확정**한다(§6 A2⁗).

2. **secret 스캔 게이트 부재가 가장 비싼 에픽에서도 미해결(A1 다섯 에픽 연속).** Epic 5는 webhook secret(5.5)·Agent token revoke/rotate(5.8)·Admin MFA·IP allowlist(5.8)·DB 연결문자열(5.2)·플랫폼 계정 token ref(5.11)로 **민감값 표면이 누적 최대**였다. 평문 누출 0은 per-story redact 어서션·`*_ref` seam·평문 거부로 지켰으나, **차단 게이트가 없어 11개 스토리 × 수작업/테스트별 점검에 의존**. 통제면이 완성돼 실제 운영 배포가 가까워질수록 게이트 부재 비용은 더 커진다.

3. **server 통제면은 완성됐으나 "자율 dispatch 런타임"은 미배선 — 가장 중요한 "보임 ≠ 동작" 경계(A8 지속).** Epic 5는 보기·하기·지키기·관측을 완성했지만, **수집→렌더→전송을 스스로 도는 루프는 없다.** `crawl_worker.py` 미존재, `migration/cutover.py`의 dispatch 루프 미코딩, 현재 유일한 실 발송 chokepoint는 운영자 주도 `test_send` 하나다. 5.10이 kill switch를 `test_send`에만 배선하며 "중앙 루프 추가 시 동일 게이트 의무"를 코드주석으로 남겼고, `verify_channel`의 실 test send는 미게이트로 남았다. **"Admin에서 다 보이고 버튼이 다 있다"가 "서비스가 자동으로 돈다"로 오인될 위험** — 다음 단계(cutover/워커)에서 명시 추적 필요.

4. **enum 회피(plain-string) 전략의 직렬화 부채가 server 경계에서 본격 표면화(A4 미완).** Epic 4가 키운 동형-유사명(`HELD`/`ACTIVE`/`USER_ACTION_*`)에 더해, Epic 5는 "Agent lowercase status → 서버 UPPER_SNAKE"(5.3)를 **API 경계마다 개별 매핑**으로 처리했다. plain-string + 경계별 변환으로 동작은 했지만, A4가 권고한 "단일 정본·단일 변환 헬퍼"는 끝내 부재 — 직렬화/매핑 로직이 여러 라우트·서비스에 분산돼 있어 정본 부재 비용이 누적 중이다(현재는 각 count-lock 테스트가 안전망).

5. **PG-gated 테스트가 무-Postgres 환경에서 광범위 skip — 일부 경로는 실 DB 미검증.** WSL/local venv에 Postgres가 없어 51건이 skip된다(2028 중 51). 5.2 3-tier·5.3 SKIP LOCKED·5.4 조건부 UPDATE race·5.6 N+1/inner join·5.11 cross-tenant INSERT 등 **"문자 충실도(literal fidelity)" 테스트가 CI/cloud Postgres 전까지 미검증**으로 이연. always-run(in-memory·offline-SQL·순수 정책)이 의미를 잠그지만, 실 DB 동시성·IntegrityError·denormalized 최적화는 Postgres CI가 붙어야 닫힌다. 메모리 `pg-gated-files-hide-pure-helpers`의 권고대로 always-run 순수 테스트는 분리됐으나, 게이트된 부분의 실측은 여전히 환경 의존.

6. **"보수적 범위 제외"가 여러 열린 질문을 후속으로 이연(계획적이나 추적 필요).** 5.7이 HELD Dispatch PG 영속을 보수적으로 보류(`get_held_dispatch→None`, docstring "Epic 3/5 reconcile")하고 dry-run/test-send 기본 seam·채널 액션을 범위 외로 제외, 5.8이 `resolve_agent_id` revoke 합성·non-sending 소비처를 LOW follow-up으로, 5.5가 `registration_code` DB 유니크 부재·타임스탬프 컬럼 미추가를 의도적 보류. 각 결정은 14표 lock·drift 최소화의 합리적 트레이드오프지만, **"배선되지 않은 정책 helper"가 누적**돼 cutover 시점에 일괄 와이어링이 필요하다.

## 5. 다음 에픽 미리보기 — MVP 완료, 이후는 Post-MVP(Epic 6/7)

**Epic 5는 MVP(P0~P4)의 마지막 에픽이다.** epics.md는 "스토리 생성 대상은 Epic 1~5(MVP)에 한정"이라 명시하며, Epic 6/7은 **Post-MVP 로드맵 항목으로만 존재(아직 스토리 미분해)**다.

- **Epic 6 (Post-MVP, P5): 온보딩과 인증 고도화** — tenant 생성·setup code·plan quota, 플랫폼 계정 다중 대상, Telegram `/register <code>` 자동 등록, Kakao 방명 검증, 배민/쿠팡 인증 흐름, ACTIVE 전 test crawl·전채널 test message 강제. **Epic 5 의존성:** 5.5(`/register` webhook·채널 검증)·5.11(엔티티 CRUD·tenant 생성)·5.7(상태 전이)·5.4(구독 게이트)가 토대를 이미 깔았다 — Epic 6은 이 primitive 위에 "강제 검증 온보딩 경험"을 얹는다.
- **Epic 7 (Post-MVP, P6): 대량 운영 안정화** — sharding·parser canary·version rollout·customer impact report. **Epic 5 의존성:** 최소 jitter·circuit breaker(FR-33)는 5.4가 이미 포함, Epic 7은 sharding·canary·rollout 고도화.

### 준비 격차 / 전환점 (MVP → 운영 cutover)

- **MVP 통제면은 완성됐으나 "자동 운영"은 cutover 미완.** §4-3·§4-6의 미배선 실체 — `crawl_worker.py`·중앙 dispatch 루프·`verify_channel` 게이트·HELD 영속·revoke resolver 합성·non-sending 소비처 — 가 **실 바인딩돼야 비로소 서비스가 자율로 돈다.** 이는 Epic 6/7 진입 전 또는 별도 cutover 워크스트림에서 명시 추적해야 한다(§6 A8-carry).
- **A4-carry(운영자 dry-run 실측)** 는 **운영 cutover 비교 직전** 여전히 필요(Epic 5에서 미해소 — 통제면만 완성, cutover 미실행).
- **A1·A2·A3 게이트 부채가 Post-MVP·운영에서 더 비싸진다.** 실 운영 배포가 시작되면 secret 표면은 운영 DB·실 token·webhook로 더 늘고, 수동 점검 비용은 비선형으로 증가한다 — Post-MVP 착수 전 또는 운영 전환 게이트로 **A1⁗·A2⁗·A3‴를 도구로 처리**하는 것이 권고.
- **PG CI 부재가 운영 신뢰도를 제한.** 51건 PG-gated skip이 닫히려면 Postgres가 붙은 CI 파이프라인이 필요하다(현재 `.github/workflows/` 부재와 동일 뿌리).

## 7. 중대 발견 / 에픽 재계획 필요 여부

**에픽 재계획 필요: 아니오.** Epic 5는 계획된 P4 범위(FastAPI 통제면·14테이블·queue·scheduler·webhook·Admin·audit·보안·7지표·부하 smoke)를 11/11 전달했고, "자율 dispatch 런타임 미배선"은 Epic 5의 처음부터의 경계(통제면 우선, cutover/워커는 후속)와 일치한다 — 발견이 계획 전제를 뒤집지 않는다. **MVP(P0~P4)가 완성**됐고, 이후는 명시적 Post-MVP(Epic 6/7) 또는 운영 cutover 워크스트림이다.

**검증된 문서 불일치(코드·git·pytest 대조 결과 — §6 A6‴의 근거):**

1. **`docs/module-architecture.md`의 "Server Domain Boundary (Epic 2–3)" 섹션이 Epic 5로 stale.** 문서는 `rider_server`를 "pure and dependency-free (no FastAPI, SQLAlchemy, or async)" / "not yet wired into a running process (the UI still calls `run_once`)" / "DB/ORM/Alembic … remain out of Epic 2–3 scope (Epic 5)"로 기술하나, Epic 5가 FastAPI app·14테이블 ORM·Alembic·async 런타임·Admin·queue·scheduler를 **실제로 올려 전부 무효**가 됐다. → **갱신 확정**(§8에서 "Cloud Server Runtime (Epic 5)" 섹션 추가 + Epic 2–3 caveat 정정).
2. **`deploy/env/backend-api.env`가 `settings.py`와 분기.** `Settings.from_env`는 `DATABASE_URL`(5.2)·`RIDER_SENDING_ENABLED`·`RIDER_ADMIN_IP_ALLOWLIST`·`RIDER_ADMIN_MFA_REQUIRED`(5.8)를 읽으나, backend-api.env(5.1 시점)는 `APP_ENV`/`APP_VERSION`/`BUILD_*`/`PORT`만 문서화 — 5.2·5.8이 추가한 server 설정 키가 누락. → **갱신 확정**(§8, 평문 secret 금지 정책 유지하며 키 보강).
3. **`_bmad-output/project-context.md`의 기술 스택·규칙이 server async 런타임 미반영.** 기술 스택 절은 `rider_server`를 "Cloud 도메인·서비스·마이그레이션(Epic 2 신설)"로만 적어 **FastAPI/SQLAlchemy 2.x(async)/Alembic/PostgreSQL** 스택과 **rider_server async 경계 규칙**(blocking sync를 async에서 직접 호출 금지, A7 가드)을 누락. Epic 4가 `rider_agent`를 반영한 것과 동형의 격차. → **갱신 확정**(§8, additive — server 스택 + async 규칙).

**코드와 일치해 갱신 폐기한 항목(검증 후 discard):**
- **`_bmad-output/planning-artifacts/architecture.md`** — job type 정본(`AUTH_CHECK`/`OPEN_AUTH_BROWSER` vs 구표기 `BAEMIN_AUTH_OPEN`)은 이미 310~313행에 정정돼 있고, "Cloud=async, Agent=sync"(336행)·FastAPI/SQLAlchemy 스택·서버 디렉터리 트리도 구현과 일치. 트리의 `crawl_worker.py`(457행)는 **의도적 계획 부분구현**(Epic 4와 동일 판정) → **폐기**(설계 의도 일치).
- **`README.md`** — `# Rider Crawl` 데스크톱 UI 앱만 안내(server/FastAPI/Admin 언급 0). server는 **아직 자율 운영 런타임이 아님**(`crawl_worker.py`·중앙 dispatch 미존재, Agent는 stub 대상, `sending_enabled` 기본 OFF). README는 **현재 출시 제품(데스크톱 exe)을 정확히 기술**하므로 server 섹션 추가는 시기상조 → **폐기**(Epic 4와 동일 판정).
- **`.env.example`** — 데스크톱 `rider_crawl` 런타임 스코프(TELEGRAM_BOT_TOKEN/CHAT_ID/THREAD_ID). server 설정은 `deploy/env/backend-api.env`·`telegram-webhook.env`에 별도 문서화(올바른 분리: .env.example=데스크톱, deploy/env=server) → **폐기**(server 설정은 deploy/env가 정본, 단 #2에서 backend-api.env 보강).

**전방 주의 2가지:**
1. **server 통제면 ≠ 자율 운영 런타임(§4-3).** `crawl_worker.py`·중앙 dispatch 루프·`verify_channel` 게이트가 미존재 — Admin에서 보이고 버튼이 있다고 "서비스가 자동으로 돈다"로 오인 금지. cutover 워크스트림에서 명시 와이어링 필요(A8-carry).
2. **A1·A2·A3 게이트 부채가 운영 전환에서 비선형 비용화.** 실 운영 배포 시 secret 표면·수치 위생·EOL 비용이 다섯 에픽 누적분 위에 더해진다 — Post-MVP/cutover 착수 전 도구로 닫는 것이 권고(A1⁗·A2⁗·A3‴·A9).

## 9. 핵심 takeaway

1. **server-only 에픽이 가능했던 건 3-패키지 단방향 위계 + Epic 2~3 도메인 0줄 재사용 덕이다.** `rider_crawl`·`rider_agent` 동시 0줄, 신규 의미 전부 `rider_server`에 가산, 14표·9-dep·enum lock 무파손 — 이 조합이 59 files/+9664, +661 passed, 무회귀 0을 만들었다. 다섯 에픽 연속 "0줄=안전마진"이 입증됐다.
2. **"코드/도구로 닫은 것은 닫혔고, 권장으로 남긴 것은 다섯 에픽 연속 안 닫혔다" — 다섯 번째 증거.** A7-carry(server async 가드)는 5.1 코드 한 번으로 11/11 자동 작동했으나, A1(secret 게이트)·A2(수치 stale)·A3(.gitattributes)는 "차단 게이트 격상"에도 또 미이행됐다. A1⁗·A2⁗·A3‴·A9는 **사람 의지가 아니라 도구(pre-commit·템플릿·CI)로** 닫아야 한다 — 이젠 권장이 아니라 운영 전환의 전제 게이트다.
3. **"보임 ≠ 동작": MVP는 통제면을 완성했지 자율 운영을 완성한 게 아니다.** Admin에서 모든 상태가 보이고 모든 버튼이 있지만, 수집→렌더→전송을 스스로 도는 `crawl_worker.py`·중앙 dispatch 루프는 아직 코드가 없다(발송은 운영자 주도 `test_send`만, kill switch 기본 OFF). cutover 워크스트림이 이 미배선 실체(A8-carry)를 채워야 비로소 서비스가 자율로 돈다 — 이 경계를 흐리면 "MVP 완성"을 "운영 가동"으로 오인하게 된다.
