## 1. 에픽 요약과 지표

Epic 4는 Epic 3까지 **런타임 미배선 순수 정책/레코드**로만 쌓아온 도메인 위에, 처음으로 **실제 실행 프로세스**(`python -m rider_agent`)를 올린 P3 에픽이다. 신규 `src/rider_agent/` 패키지가 기존 `rider_crawl`의 crawler/parser/renderer/Gmail 2FA/Kakao sender를 **단방향 import로만 재사용**하면서, 중앙 서버(Epic 5)에 **outbound-only HTTPS**로 등록·heartbeat·job claim/complete(lease)를 수행하고, BrowserProfile/CDP 격리·기대 센터검증·KakaoTalk FIFO 직렬 전송·방명 검증·배민 사람 개입형 재인증·쿠팡 Gmail 2FA 메일함 분리·lock·재부팅 autostart를 다뤘다.

Epic 3가 "`rider_crawl` 0줄로 실행 흐름 재배선"이었다면 Epic 4는 **"`rider_crawl`·`rider_server` 모두 0줄로 새 런타임 추가"** 다. 핵심 제약이 한 단계 더 강해졌다: 신규 의미를 전부 `src/rider_agent/`에 가산하되, `rider_agent`는 `rider_crawl`만 import하고 **`rider_server`는 import조차 하지 않으며**(enum 값은 plain-string으로 미러), Cloud의 async와 섞지 않는 **sync 런타임**을 유지한다. 서버 측 job 생성·queue·Admin은 Epic 5이므로, 4.x의 테스트 형태는 **서버 stub/mock·주입 seam에 대한 동작 검증**이다.

## 3. 잘된 점 (Successes)

1. **`rider_crawl`·`rider_server` 동시 0줄 변경 — 새 런타임을 추가하면서도 두 기존 패키지 완전 보존.** 9개 스토리가 `src/rider_crawl/`·`src/rider_server/`를 단 한 줄도 고치지 않고(`git diff -w 8619723..HEAD` 양쪽 빈 출력) `src/rider_agent/` 14개 모듈만 가산했다. `rider_agent`는 `rider_crawl`만 단방향 import하고 `rider_server`는 **import조차 하지 않아**(enum 값을 plain-string으로 미러) 패키지 3개의 의존 위계가 깨끗하게 유지됐다. Epic 3의 "0줄=무회귀 안전마진" 규약이 한 단계 더 강한 형태로 계승됐다.

2. **4.1 AST 가드 = 에픽 전체를 잠그는 단일 강제 장치.** 4.1이 `src/rider_agent/**/*.py`를 `rglob`로 훑어 (a) sync 전용(`async`/`await`/직접 `import asyncio` 0), (b) third-party 루트=`rider_crawl`만, (c) `rider_crawl→rider_agent`·`rider_agent→rider_server` 엣지 0, (d) `pyproject.toml` deps 정확히 9개·핀 고정을 단언했다. 4.2~4.9는 **새 가드를 쓰지 않고 4.1의 가드를 상속**만 했다 — "가드는 한 번 짜고 물려받는다"가 완벽히 작동. 이 가드가 매번 막아낸 유혹: `pywin32`/`psutil`/`requests` 추가.

3. **stdlib-only로 의존성 0 증가 — 가드와 배포 단순성을 동시에 지킴.** HTTPS=`urllib`(텔레그램 선례), DPAPI=`ctypes`/crypt32, 주기 루프=`threading`/`time`, 포트할당=`socket`. `pywin32`/`win32crypt`/`psutil`/`requests`를 의도적으로 회피해 4.1의 "deps 9개" lock과 "third-party 루트=rider_crawl" 가드를 9/9 유지했고, PyInstaller onefile 배포 표면도 키우지 않았다.

4. **fail-closed가 Agent 전 경로 관통(오발송보다 미발송·미실행).** 4.4 claim한 job만 실행 + lease 자가점검 → 4.5 포트/프로필 중복·CDP 충돌·센터 불일치 시 작업 미시작(`TARGET_VALIDATION_FAILURE`) → 4.6 방명 모호·창 확인 실패 시 임의 전송 금지(`KAKAO_FAILURE`, 다른 방 자동복구 금지) → 4.7 Session 0이면 Kakao worker 미기동(interactive gate) → 4.8 비ACTIVE면 메시지 생성·전송 0, OTP 우회 0, timeout 시 무한재시도 금지 → 4.9 자동복구 실패 시 반복 인증요청 중단. 안전 기본값이 한 스토리도 빠짐없이 적용됐다.

5. **"테스트 통과 ≠ 격리 증명"을 잡아낸 4.9 회귀 트랩 — 에픽 최고 난이도 판단.** 쿠팡 재사용 기본값은 **단일 공유 파일**(`config.gmail_token_path`)을 읽어, `recover`를 기본값에 두면 **운영에서 모든 고객이 같은 token 파일을 공유**(ADD-15 위반)하지만 **fake 주입 테스트는 전부 통과해 회귀를 숨긴다.** `build_coupang_recover`가 `dataclasses.replace`로 메일함별 token 경로를 분기하고, spy로 "메일함마다 경로가 갈라진다"를 명시 단언(`test_build_coupang_recover_wires_distinct_token_path_per_mailbox`)해 막았다. frozen `AppConfig` 덕에 **0줄 소스 변경**으로 격리를 와이어링.

6. **메모리 자산이 코딩 규약으로 내재화 — 같은 함정을 사전 회피.** `enum-member-count-locks`(plain-string으로 9/9 회피), `redact-skips-operational-ids`(4.6 room_name·4.9 mailbox ref를 해시/고정문자열로 — 운영 ID 누출 0), `negative-guard-tests-use-ast`(4.1·4.8 import-엣지), `crlf-roundtrip-idempotency`(4.7), `baemin-no-action-required-signal`(4.8 — `MissingPerformanceDataError`→AUTH_REQUIRED 광역 매핑 금지, 주입 probe), `coupang-gmail-2fa-state-vocab`(4.9 — `USER_ACTION_REQUIRED`≠`USER_ACTION_PENDING`), `agent-main-runpy-warning`(import 함수 내부 지연)이 모두 실 사용. **8개 메모리가 9개 스토리에서 적중**했다.

7. **"primitive vs 와이어링" 소유 규율 일관 — 빈 stub 0개.** 매 스토리가 primitive만 내고 와이어링은 명시 이연(4.3 heartbeat reporter→4.4 thread 기동, 4.4 execute seam→4.5/4.6/4.8 워커, 4.5가 4.4의 "CRAWL 워커는 4.5" 전방 포인터를 **중간에 정정**). 빈 stub 파일 생성은 일관 금지였고, 9/9 리뷰에서 `[x]` 태스크의 **허위 완료 0건**이 확인됐다. heartbeat provider seam 3개(active_jobs/browser_profiles/kakao_status)를 후속 3스토리가 정확히 하나씩 채우는 동안 `heartbeat.py`는 0줄 유지.

## 4. 어려웠던 점 / 반복 패턴 (Challenges — 무비난, 시스템 관점)

1. **테스트 수치 stale — 9/9 전수 재발(Epic 1 A2 → 2 → 3 → 4, 네 에픽 연속).** 모든 스토리에서 dev-story 시점 기록 수치가 이후 qa-generate-e2e 케이스 추가로 낡았고, 리뷰가 매번 재측정해 정정(MEDIUM 9건 전부 이 유형). 근본 원인은 변함없이 시스템적이다: `dev-story → QA gap-fill → review` 순서에서 중간에 적는 절대 수치는 **항상 잠정값**이다. A2″가 "dev 수치 기재란 물리 제거"를 실제 템플릿에 반영하지 않아 구조적으로 매번 발생한다. **9회 연속은 더 이상 "주의로 고칠 문제"가 아니라 "도구로 닫아야 할 문제"임을 확정**한다(§6 A2‴).

2. **secret 스캔 게이트 부재가 가장 비싼 에픽에서도 미해결(A1″ 4 에픽 연속).** Epic 4는 Agent token(4.2)·heartbeat Bearer 헤더(4.3)·job 이벤트(4.4)·Gmail OAuth/OTP/쿠팡 비밀번호(4.9)로 **민감값 표면이 에픽 중 최대**였다. 평문 누출 0은 수동 grep·fake-value·`__repr__` 마스킹·redact 어서션으로 지켰으나, **차단 게이트가 없어 9개 스토리 × 수작업 점검에 의존**했다. Epic 5에서 FastAPI/DB/secret 와이어링이 본격화되면 표면이 또 늘어난다 — 게이트 비용은 갈수록 커진다.

3. **File List drift 반복(4.5 MEDIUM, 4.4/4.7/4.8 LOW).** qa-generate-e2e가 만든 `test-summary-*.md`(untracked, `_bmad-output/` 하위)와 4.5의 `test_agent_package.py`(reuse identity 단언 추가분)가 File List에 누락. A2″의 수치 stale과 동일 뿌리 — dev 시점 산출물 목록이 QA 보강분을 반영 못 함. `_bmad-output/` 하위는 리뷰 제외라 대부분 LOW로 흡수됐으나, 4.5처럼 `tests/` 실코드가 빠지면 MEDIUM이 된다.

4. **A3′(.gitattributes) 미이행의 비용이 4.7에서 실제 발생.** 세 에픽 연속 "비용이 작아 밀린" A3′가 이번엔 **CRLF round-trip idempotency 버그**로 터졌다(Startup `.cmd` 리터럴 `\r\n` → text-mode 재변환 → 재등록마다 rewrite). 코드로는 `\n` 재작성으로 막았으나, EOL 정규화가 게이트였다면 클래스 자체가 없었을 버그다. "비용이 작다"는 판단이 누적 위험을 가린 사례.

5. **placeholder probe·미존재 워커가 Epic 5로 다량 이연(계획된 부분구현이나 추적 필요).** 4.8의 3개 default probe(`default_login_probe`/`default_open_auth_browser`/`default_detect_completion`)와 4.9의 `is_reauth` predicate는 **Windows-gated 비기능 placeholder**(실제 바인딩은 운영/Epic 5). 4.5/4.8/4.9가 가리키는 `workers/crawl_worker.py`(CRAWL_* 수집/Snapshot 업로드 실행기)는 **아직 존재하지 않는다**(architecture.md:454 계획). primitive는 완성됐으나 "실제로 동작하는 end-to-end Agent"는 Epic 5 서버·워커 와이어링 전까지 미완 — Epic 5 진입 시 이 목록을 명시 추적해야 한다(§7).

6. **enum 회피 전략이 만든 "동형-유사명" 어휘 부채 누적.** Epic 3가 남긴 `HELD`(전송 보류) vs `HELD`(구독 게이트) 동명-다른타입에 더해, Epic 4가 `USER_ACTION_REQUIRED`(쿠팡 4.9) vs `USER_ACTION_PENDING`(배민) 유사명, 그리고 job-type 어휘 분기(`AUTH_CHECK`/`OPEN_AUTH_BROWSER` 정본 vs architecture.md의 `BAEMIN_AUTH_OPEN` 구표기)를 추가했다. plain-string + 값 미러로 lock은 안전하게 피했지만, **plain-string 미러가 늘수록 "어느 것이 정본인가"가 Epic 5 직렬화/DB 와이어링에서 비용**이 된다 — A4의 단일 정본 필요성을 키웠다.

## 7. 중대 발견 / 에픽 재계획 필요 여부

**에픽 재계획 필요: 아니오.** Epic 4의 발견은 Epic 5의 계획 전제를 뒤집지 않는다. FastAPI 서버·PostgreSQL·queue·scheduler·Admin 방향은 그대로 유효하고, 4.x가 만든 Agent 클라이언트 계약(register/heartbeat/claim/complete/events)·lease 협조 인지·`browser_profiles`/auth_state/mailbox ref 데이터가 Epic 5 서버가 채울 자리를 정확히 비워뒀다.

**검증된 문서 불일치(코드·git·pytest 대조 결과 — §6 A6″의 근거):**

## 9. 핵심 takeaway

1. **3 패키지 의존 위계(`rider_agent→rider_crawl` 단방향, `rider_server` 미참조)가 새 런타임 추가의 안전마진이었다.** `rider_crawl`·`rider_server` 동시 0줄 + stdlib-only + 4.1 AST 가드 상속 — 이 조합이 9개 스토리, +322 테스트, 무회귀 0을 만들었다. Epic 5 server도 같은 규율(단 server는 async 정본·별도 가드)로 가야 한다.
2. **"게이트/도구/코드로 닫은 것은 닫혔고, 권장으로 남긴 것은 4 에픽 연속 안 닫혔다."** A7(가드)은 4.1 코드+메모리로 자동 작동했으나, A1·A2·A3는 "차단 게이트로 격상"이라는 최강 표현에도 미이행돼 A2는 9/9·A1은 4 에픽·A3는 4.7에서 실 버그로 비용화됐다. A1‴(pre-commit), A2‴(qa-e2e 자동 기입), A3″(.gitattributes)는 **사람 의지가 아니라 도구로** 닫아야 한다.
3. **"테스트 통과 ≠ 격리/정확성 증명"이 Agent 에픽의 가장 비싼 교훈이었다.** 4.9 회귀 트랩(fake 테스트는 통과하나 운영에선 token 공유)을 spy 단언으로, 4.6 운영 ID 누출을 고정문자열로, 4.8 배민 인증을 주입 probe로(파서 오류를 인증 오류로 오인 금지) 막았다 — 외부 부작용을 가진 런타임에서는 "통과"가 아니라 "격리·우회불가·미오인"을 명시 단언해야 한다.
