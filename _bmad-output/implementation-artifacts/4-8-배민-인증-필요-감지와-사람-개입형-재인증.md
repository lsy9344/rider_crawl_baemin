## Story

As a 운영자,
I want **배민 수집/인증 점검 중 휴대폰 인증·로그인 만료를 감지해 작업을 `AUTH_REQUIRED` 상태로 두고 실적 메시지를 만들거나 보내지 않으며(어떤 고객/대상/브라우저 프로필이 인증을 요구하는지 서버에 보고)**, **사람이 해당 프로필 브라우저를 인증 모드로 열어 휴대폰 인증을 완료한 뒤에만(시스템은 인증번호를 취득·우회·자동 통과하지 않고 사람이 완료한 `AUTH_VERIFIED` 상태만 감지) 작업을 재개**하고, **인증이 제때 완료되지 않으면 대상이 `AUTH_REQUIRED`(또는 auth timeout)로 유지되고 전송은 진행하지 않으며 무한 재시도하지 않게(인증 실패/timeout이 운영 상태에 남음)**, 이 모든 것을 **OS·브라우저·시간 부작용을 전부 주입 가능하게 한 순수 동기 primitive 신규 서브패키지 `src/rider_agent/auth/`(`baemin_auth.py` — auth 상태 분류기 + `AUTH_CHECK`/`OPEN_AUTH_BROWSER` 실행자 + bounded 재인증 대기 + `build_auth_execute_job` 라우터)** 로 갖고 싶다,
so that 인증을 우회하지 않으면서(ADD-15·ops 금지) 인증으로 막힌 대상에 잘못된 메시지를 보내지 않고(NFR-2 fail-closed), 사람 확인 후 안전하게 재개하며, 무한 재시도로 운영을 망가뜨리지 않는다(NFR-4) (P3, FR-17·18, NFR-2·4·5·8, ADD-15).

> **이 스토리의 성격 — "배민 auth 상태 감지 + 사람 개입형 재인증 실행자 primitive"만. Epic 4 플랫폼-인증 시리즈(4.8 배민 / 4.9 쿠팡 Gmail 2FA)의 첫 스토리.** 4.1~4.7이 만든 토대(reuse seam·job_loop execute_job 주입점·BrowserProfileManager·heartbeat capability) 위에 **(a) 배민 auth 상태 분류기**(`BrowserActionRequiredError`/주입 login probe → `AUTH_REQUIRED`/`ACTIVE`, fail-closed), **(b) `AUTH_CHECK` 실행자**(로그인 상태만 점검·보고, 수집/전송 안 함), **(c) `OPEN_AUTH_BROWSER` 실행자**(프로필 브라우저를 사람용으로 열고 **사람이 완료한 인증만 감지** → `AUTH_VERIFIED`, OTP 취득·자동입력·우회 0), **(d) bounded 재인증 대기/timeout**(무한 재시도 금지 → `AUTH_REQUIRED`/auth timeout)을 얹는다. **실제 KakaoTalk/Chrome/휴대폰 인증·실 브라우저·실 시계는 본 스토리가 한 줄도 "테스트에서" 호출하지 않는다** — 전부 주입 fake(login probe·open_browser·detect_completion·now·sleep·transport)로 결정적 검증한다. ## Acceptance Criteria

**AC1 — 배민 수집/점검 중 인증 필요 감지 → AUTH_REQUIRED·메시지 미생성·서버 보고 (FR-17, NFR-2)**

1. **Given** 배민 수집 중 휴대폰 인증 또는 로그인 만료가 필요할 때 **When** Agent 가 인증 필요 상태를 감지하면 **Then** 작업이 **`AUTH_REQUIRED` 상태로 전환되고 실적 메시지를 생성·전송하지 않는다(fail-closed)**: `auth/baemin_auth.py` 가 분류기 `classify_baemin_auth_state(*, snapshot_ok=None, error=None) -> str` 를 제공한다 — `BrowserActionRequiredError`(reuse seam) → `AUTH_REQUIRED`; 정상 달성현황 snapshot(`snapshot_ok=True`) → `ACTIVE`; 그 외/모호는 `UNKNOWN`(또는 비-auth 예외는 분류하지 않고 호출자에 위임). `AUTH_CHECK` 실행자는 **수집/렌더/전송을 호출하지 않는다**(로그인 상태만 점검). [Source: src/rider_agent/reuse.py(29-35·69), src/rider_agent/browser_profile.py(63·373-377), architecture-contract.md(126), epics.md AC(862-864)]
2. **And** **`AUTH_CHECK` job 실행자가 로그인 상태만 점검해 `AUTH_REQUIRED` 또는 `ACTIVE` 를 보고한다**: `execute_auth_check_job(job, *, login_probe, now=time.time, log=None) -> JobResult` — 주입 `login_probe(job) -> str`(auth_state 평문 상수 반환, 기본은 Windows-gated lazy probe)로 상태를 얻어, `ACTIVE` 면 `make_success_result(result_json={target_id, auth_state})`, `AUTH_REQUIRED` 면 **메시지 생성 없이** auth-required 를 표면화한다(`error_code=AUTH_REQUIRED` 실패 결과 **또는** `auth_state=AUTH_REQUIRED` 성공 결과 + auth-required 이벤트 — 둘 중 하나로 일관). 어떤 **고객/대상/브라우저 프로필**이 인증을 요구하는지 `target_id` + profile **id/ref** 로 보고한다(**raw 경로/secret 0**). [Source: src/rider_agent/job_loop.py(175-228), src/rider_agent/browser_profile.py(396-414), epics.md AC(865), architecture-contract.md(126)]
3. **And** 분류/실행자는 **순수 동기·주입 가능**(테스트에서 실 브라우저/실 crawl 미호출)하고 **fail-closed**(인증 필요면 실적 메시지 0)이며 **secret/OTP/휴대폰/raw 경로를 노출하지 않는다**(result_json·event·log 에 `target_id`/평문 auth_state/ref 만). [Source: project-context.md(36·81·87), operations-security-test-contract.md(15·93), src/rider_agent/workers/kakao_sender.py(89-98)]

**AC2 — 사람 개입형 재인증: 프로필 열기 + 사람-완료 감지만, OTP 취득·우회 0 (FR-18, ADD-15)**

### 열린 질문 / 의도된 부분 구현 (반드시 읽을 것)

- **배민 auth-required 의 "기본 real probe" 정책 — `rider_crawl` 0줄 유지.** 배민 크롤러는 **`BrowserActionRequiredError` 를 어디서도 raise 하지 않는다**(쿠팡만 raise; 배민은 로그인 안 됨이 `MissingPerformanceDataError`/`RuntimeError` 로 표면화 — memory: baemin-no-action-required-signal). 따라서 **`MissingPerformanceDataError`→`AUTH_REQUIRED` 광범위 매핑은 금지**(파서 문제를 인증 문제로 오인). 권장: auth-required 감지는 **주입 `login_probe`/`detect_completion`** 으로 흡수하고, 기본 real probe 는 (a) `BrowserActionRequiredError`(prepare/reuse seam) 소비 + (b) read-only 달성현황 도달 휴리스틱(`_looks_like_baemin_achievement_report` 는 **private** 이라 직접 import 대신 동일 정책을 probe 안에서 적용하거나 `crawl_snapshot` 성공/예외로 판정)로 둔다. **핵심: `rider_crawl` 에 배민 login-page 감지 코드를 새로 넣지 않는다**(0줄·안전 마진). 실제 real-probe 의 정밀화가 필요하면 별도 스토리로 분리. [Source: memory/baemin-no-action-required-signal, src/rider_crawl/crawler.py(40-42·737-744), src/rider_agent/reuse.py(24·69)]
- **`AUTH_CHECK` 결과 형태: failure(`error_code=AUTH_REQUIRED`) vs success(`auth_state=AUTH_REQUIRED`).** 계약은 "report AUTH_REQUIRED or ACTIVE"(상태 보고)다. 본 스토리는 **둘 중 하나로 일관**하면 된다 — 권장: `AUTH_REQUIRED` 도 `make_success_result(result_json={auth_state:AUTH_REQUIRED, target_id})` + auth-required **이벤트**로 보고(상태 점검은 "실패"가 아니라 "필요 신호"). 단, 실제 `CRAWL_BAEMIN` 경로에서 인증으로 막혀 수집이 중단될 때는 `make_failure_result(ERROR_AUTH_REQUIRED, ...)`(`FailureCategory.AUTH_REQUIRED`)가 맞다. 두 경로의 어휘(`AUTH_REQUIRED`)는 같고 표면(result_json vs error_code)만 맥락에 맞춘다 — 본 스토리는 `AUTH_CHECK` 표면을 정하고, crawl 경로 failure 매핑은 분류기 + `make_failure_result` 헬퍼 재사용으로 후속(crawl 워커)이 소비. [Source: architecture-contract.md(124·126), src/rider_agent/job_loop.py(175-207), src/rider_server/domain/states.py(180)]
- **job-type 명 변이: `AUTH_CHECK`/`OPEN_AUTH_BROWSER`(contract·heartbeat) vs `BAEMIN_AUTH_OPEN`(architecture.md:310).** heartbeat.py(64-65)·architecture-contract.md(126-127)의 **`AUTH_CHECK`/`OPEN_AUTH_BROWSER` 를 정본**으로 쓴다(capability 상수가 이미 그 이름). `BAEMIN_AUTH_OPEN` 은 architecture.md 의 구버전 표기 — 평문 상수라 후속이 별칭을 늘려도 lock 0. [Source: src/rider_agent/heartbeat.py(64-65), architecture-contract.md(126-127), architecture.md(310), memory/agent-job-type-vocab]
- **재인증 대기 상한 값.** `max_wait_seconds`/`poll_interval_seconds`/`max_attempts` 의 운영 기본값은 4.5 `recover_profile`(attempts 3·backoff 1.0) 규모를 본떠 모듈 상수로 두되, **테스트는 항상 작은 값/주입 `now`·`sleep`** 로 결정적 검증한다. 운영 실값은 Epic 5 스케줄/lease(4.4)와 정합하게 조정 가능(본 스토리는 bounded 보장만).
