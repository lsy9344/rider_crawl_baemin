# UI/UX 설정 편의성 연구

작성일: 2026-06-19  
대상: Rider Result Monitoring 웹 관리자 UI와 로컬 설정 UI  
목적: 목표 웹앱의 사용자 편의성, 특히 고객/계정/전송 설정이 쉽게 되어 있는지 검토하고 개선할 부분을 찾는다.

## 1. 결론

현재 제품은 운영 도구로 필요한 기능을 많이 갖추고 있지만, 설정 경험은 아직 "마법사형 온보딩"보다 "관리자가 내부 구조를 이해하고 직접 채우는 폼"에 가깝다.

가장 큰 문제는 세팅 흐름이 사용자의 실제 순서와 완전히 맞지 않는 점이다. 사용자는 보통 고객 생성, 플랫폼 계정 연결, 업체 등록, 전송 채널 연결, 테스트 수집, 실발송 켜기 순서로 진행한다. 그런데 현재 화면은 관리 탭 안에 여러 fieldset을 나열하고, 일부 고급 설정은 접힘 안에 들어가 있어 첫 사용자에게 다음 행동이 바로 보이지 않는다.

따라서 1순위 개선은 새 기능 추가가 아니라 설정 흐름을 한 번의 안내된 절차로 재구성하는 것이다. "고객 추가 마법사"를 만들고, 각 단계마다 필요한 값만 보여주며, 마지막에 테스트 수집과 테스트 전송까지 끝내도록 해야 한다.

## 2. 검토 범위와 근거

검토한 주요 파일:

| 경로 | 본 내용 |
| --- | --- |
| `src/rider_server/admin/templates/dashboard.html` | 관리자 웹 대시보드, 모니터링/관리 탭, 스타일, drawer, 필터 |
| `src/rider_server/admin/templates/_entity_admin.html` | 고객, 플랫폼 계정, 업체, 전송 채널, 텔레그램 설정 폼 |
| `src/rider_crawl/ui.py` | 기존 Tkinter 설정 UI, 탭별 크롤링 설정, 검증, 시작/중지 흐름 |
| `src/rider_crawl/ui_settings.py` | UI 설정 기본값, 비밀값 분리 저장, 탭별 기본 설정 |
| `README.md` | 설치, Chrome 준비, UI 설정, 서버/DB 실행 설명 |
| `_bmad-output/project-context.md` | 프로젝트 제약, 보안, 운영 규칙 |

실행 확인:

- `http://127.0.0.1:8017/health`는 정상 응답했다.
- `/admin` 화면은 현재 로컬 환경에서 500으로 실패했다. 원인은 `DATABASE_URL`이 잡힌 상태에서 PostgreSQL 연결이 거부된 것이다. trace는 `src/rider_server/admin/dashboard_repository_postgres.py`의 dashboard repository 경로에서 DB 연결 실패로 확인됐다.
- 그래서 이번 평가는 실제 렌더 화면 전체가 아니라 템플릿, 코드 흐름, 자동 탐지 결과를 기준으로 한다.

자동 탐지:

- `npx impeccable --json --fast src/rider_server/admin/templates src/rider_crawl/ui.py` 실행.
- `dashboard.html`에서 작은 글자, 촘촘한 타입 계층, Geist Mono 사용, body copy의 em dash 과다를 경고로 탐지했다.
- `npx --no-install impeccable --json src/rider_server/admin/templates`도 같은 계열의 경고를 냈다. 주요 항목은 tiny text, Geist Mono 과사용, flat type hierarchy, em dash 과다다.
- `uv run python -m pytest -q tests/server/test_admin_dashboard.py tests/server/test_admin_entity_crud.py tests/test_ui_helpers.py tests/test_ui_settings.py`는 281개 통과했다. 보조 평가에서 직접 `python -m pytest` 경로는 Python site import 중 `UnicodeDecodeError`가 나서 이 환경에서는 `uv run` 경로가 더 안정적이라고 확인했다.

### 2.1 정량 진단 요약

이번 평가는 `impeccable`의 제품 UI 기준을 적용했다. 이 앱은 마케팅 페이지가 아니라 운영자가 반복해서 쓰는 업무 도구이므로, 멋보다 예측 가능성, 실수 방지, 빠른 회복이 더 중요하다.

## 3. 현재 설정 흐름

### 3.1 관리자 웹 UI

관리자 UI는 `/admin` 안에서 모니터링과 관리를 탭으로 나눈다.

주요 구조:

- 모니터링 탭: 업체 상태, 검색, 플랫폼 필터, 인증 필요 필터, 상세 drawer.
- 관리 탭: 엔티티 관리.
- 엔티티 관리의 상위 단계:
  1. 새 업체 추가
  2. 전송 연결
  3. 등록된 업체 편집/비활성화
- 접힌 보조 영역:
  - 구독 관리
  - 플랫폼 계정
  - 메시지 채널
  - 고객(Tenant)
  - 고객별 텔레그램 설정

좋은 점:

- `_entity_admin.html`은 이미 raw id 직접 입력을 줄이고 `<select>`를 사용하려는 방향이다.
- "① 새 업체 추가", "② 전송 연결", "③ 등록된 업체"처럼 단계 번호가 있다.
- 고객 전체 보기에서는 작업 고객을 먼저 선택하라는 경고가 있다.
- 비활성화, 삭제, 위험 액션에는 confirm을 둔다.
- 모니터링 화면에는 검색, 플랫폼 필터, 인증 필요 필터가 있다.

불편한 점:

- 실제 새 고객 세팅의 첫 단계인 "고객 생성"과 "플랫폼 계정 생성"이 접힌 보조 영역 안에 있다.
- 새 업체 추가는 첫 번째로 보이지만, 업체를 만들려면 플랫폼 계정이 먼저 있어야 한다.
- 텔레그램 봇 토큰, webhook secret, 실발송 토글이 고객별 설정에 따로 있고, 전송 채널 생성은 다른 fieldset에 있다.
- 사용자에게 "현재 몇 단계 중 몇 단계인지"가 보이지 않는다.
- `BAEMIN`, `COUPANG`, `PAYMENT_ACTIVE`, `SETUP_PENDING`, `sending_enabled`, `webhook secret` 같은 내부 용어가 화면에 그대로 나온다.

### 3.2 로컬 Tkinter 설정 UI

로컬 UI는 `크롤링1`부터 `크롤링9`까지 탭을 만든다. 각 탭에 20개가량의 입력 필드가 있고, 아래쪽에 플랫폼, 브라우저 연결, 전송 방식, 자동복구 체크가 있다.

좋은 점:

- 플랫폼 선택에 따라 쿠팡 URL이 `peak-dashboard`로 자동 고정된다.
- 텔레그램/카카오톡 선택에 따라 관련 필드만 활성화한다.
- 메시지 전송 기본값이 꺼져 있어 처음부터 실발송하는 위험을 줄인다.
- 시작 전 확인 문구가 운영 실수를 많이 막는다.
- 저장 시 CDP 주소, 배민 센터명, 쿠팡 기대 센터명, 텔레그램 필수값, 카카오 필수값을 검증한다.

불편한 점:

- 한 탭 안에 URL, 센터명, CDP, 브라우저 프로필, 토큰, 채팅방, 로그, timeout, 쿠팡 로그인, 인증 이메일이 모두 보인다.
- 초보자가 "처음에 뭘 입력해야 하는지"보다 "전체 설정 목록"을 먼저 만난다.
- 시작 전 확인이 11개 문장으로 길어, 실제 조작 중에는 읽히기 어렵다.
- `보조 URL`, `락 타임아웃`, `CDP 주소`, `앱 전용 브라우저 프로필 경로` 같은 고급 항목이 기본 화면에 노출된다.
- 쿠팡에서 `센터명(배민 센터명 / 쿠팡 기대 센터·상점명)`처럼 한 필드를 두 의미로 쓰는 점은 정확하지만 인지 부담이 크다.

## 4. 사용성 진단

### 4.1 설정은 기능적으로는 가능하지만 쉽게 되어 있지는 않다

현재 설정은 숙련 운영자가 문서를 읽고 진행하면 가능하다. 하지만 새 고객을 빠르게 추가하는 도구라고 보기에는 단계가 흩어져 있다.

예를 들어 새 고객을 추가하려면 관리자가 다음 개념을 알아야 한다.

- Tenant와 고객의 관계
- 플랫폼 계정과 업체의 관계
- 업체와 메시지 채널의 관계
- 전송 규칙의 의미
- 텔레그램 채널과 고객별 텔레그램 봇 토큰 설정의 차이
- 실발송 게이트가 기본 OFF라는 점

이 개념들은 시스템 설계에는 맞지만, 첫 세팅 화면에서는 너무 많이 보인다.

### 4.2 설정 오류 방지는 꽤 좋다

검증 로직은 강하다.

- CDP는 로컬 주소만 허용한다.
- 배민은 센터명 또는 센터 ID를 요구한다.
- 같은 배민 센터명을 쓰는 탭에는 센터 ID를 요구한다.
- 쿠팡은 `https://partner.coupangeats.com/page/peak-dashboard` 형식을 강제한다.
- 쿠팡 기대 센터/상점명을 필수로 요구한다.
- 쿠팡 자동복구를 켜면 로그인 ID, 비밀번호, 인증 이메일, 앱 비밀번호를 요구한다.
- 텔레그램 전송을 켜면 봇 토큰과 채팅방 ID를 요구한다.
- 카카오 전송을 켜면 채팅방명을 요구한다.

이 점은 오발송과 잘못된 크롤링을 줄이는 데 중요하다. 개선할 때도 이 검증은 유지해야 한다.

### 4.3 화면 언어가 운영자 친화적이지 않은 부분이 있다

일부 라벨은 개발자나 내부 관리자에게는 정확하지만, 비개발 운영자에게는 어렵다.

예:

- `Tenant`
- `external_id`
- `webhook secret`
- `registration_code`
- `sending_enabled`
- `PAYMENT_ACTIVE`
- `SETUP_PENDING`
- `soft delete`
- `fail-closed`

운영 화면에서는 내부 용어를 그대로 보여주기보다, 쉬운 이름과 보조 설명을 써야 한다. 내부 값은 필요하면 괄호나 tooltip로만 보여주는 편이 좋다.

### 4.4 시각 디자인은 업무용 도구로는 방향이 맞지만 글자가 작다

관리자 웹 UI는 다크 우선, 카드/테이블/필터/상세 drawer 조합으로 업무 도구에 맞는 구조다. 다만 `dashboard.html`에는 `0.68rem`, `0.7rem`, `0.72rem`, `0.74rem` 같은 작은 글자가 많다.

자동 탐지는 10.56px, 11.84px 수준의 작은 body text를 잡았다. 운영자는 장시간 모니터링하고 실수를 줄여야 하므로, 설정과 위험 액션 화면의 기본 글자는 14px 이상으로 두는 편이 좋다.

또한 전체 UI가 Geist Mono 중심이다. 숫자와 로그에는 monospace가 좋지만, 한국어 설명과 폼 라벨까지 monospace면 읽는 속도가 떨어진다. 본문과 라벨은 system sans 또는 Pretendard 계열, 숫자/ID만 mono로 나누는 편이 더 좋다.

### 4.5 첫 화면 성공 경험이 약하다

로컬 UI는 `1회 실행`과 `메시지 전송 끄기`를 안내한다. 이 방향은 좋다.

관리자 웹 UI에는 새 고객을 추가한 뒤 "테스트 수집", "테스트 메시지", "실발송 ON"까지 이어지는 완료 경험이 약하다. 세팅이 끝났는지 판단할 체크리스트가 화면에 없다.

설정 UX는 "값을 저장했다"보다 "이 고객은 수집과 전송 테스트를 통과했다"가 끝점이어야 한다.

## 5. 우선순위 이슈

### P0. `/admin`이 DB 연결 실패 시 화면 전체가 500으로 죽는다

왜 중요한가:

- 관리자 화면을 열 수 없으면 설정 UX 이전에 운영이 막힌다.
- 로컬 리뷰에서 `DATABASE_URL`이 잡혀 있지만 DB가 꺼진 상태라 `/admin`이 500을 반환했다.
- 운영에서도 DB 장애가 있을 때 최소한 읽을 수 있는 오류 화면, 점검 안내, 재시도 상태가 필요하다.

개선 방향:

- `/admin` 렌더 중 DB 연결 실패를 잡아 "DB 연결 실패, 설정 확인 필요" 화면을 보여준다.
- health/readiness를 분리한다.
- 로컬 개발 실행은 `.env`의 `DATABASE_URL` 영향으로 원치 않게 PostgreSQL 경로를 타지 않도록 실행 가이드를 보강한다.

### P1. 새 고객 세팅 절차가 한 화면 흐름으로 묶여 있지 않다

왜 중요한가:

- 새 고객 세팅은 제품의 핵심 업무다.
- 현재는 고객, 플랫폼 계정, 업체, 채널, 전송 규칙, 텔레그램 설정이 흩어져 있다.
- 관리자가 순서를 틀리면 빈 드롭다운, 비활성 버튼, "대상 선택 필요" 같은 결과를 만난다.

개선 방향:

- `고객 추가 마법사`를 만든다.
- 단계는 고객 정보, 플랫폼 계정, 업체/센터, 전송 채널, 테스트, 실발송 확인으로 나눈다.
- 각 단계는 이전 단계가 끝나야 다음으로 간다.
- 마지막 화면에 "세팅 완료" 체크리스트를 보여준다.

### P1. 기본 화면에 고급 설정이 너무 많이 노출된다

왜 중요한가:

- CDP, 프로필 경로, timeout, lock, webhook, registration code는 초보자가 첫 세팅 때 만지면 위험하다.
- 값은 필요하지만, 처음부터 보이면 판단 부담이 커진다.

개선 방향:

- 기본 설정과 고급 설정을 나눈다.
- 기본 설정에는 플랫폼, 고객명, 센터/상점명, 전송 채널, 테스트만 둔다.
- CDP/프로필/timeout/lock/webhook/registration code는 고급 섹션에 둔다.

### P1. 비밀값과 실발송 게이트가 전송 채널 설정과 분리되어 있다

왜 중요한가:

- 운영자는 "텔레그램으로 받을 방을 연결했다"와 "이 고객의 봇 토큰, webhook 보안키, 실발송 허용이 준비됐다"를 하나의 작업으로 이해한다.
- 현재 메시지 채널 생성은 `_entity_admin.html:184` 근처에 있고, 고객별 텔레그램 봇 토큰과 `sending_enabled`는 `_entity_admin.html:247` 근처에 따로 있다.
- 플랫폼 계정 비밀번호도 `_entity_admin.html:149`에서 DB 직접 저장이라고만 보이며, 설정됨/교체/삭제 같은 안전한 상태 표현이 약하다.
- 이 구조는 채널은 만들었지만 실발송 게이트가 꺼져 있거나, 토큰은 바꿨지만 테스트 전송을 안 한 상태를 만들기 쉽다.

개선 방향:

- 전송 채널 단계 안에서 `채널 식별자`, `고객별 봇 토큰`, `webhook 보안키`, `실발송 상태`를 한 번에 확인하게 한다.
- 비밀값 입력란은 실제 값을 다시 보여주지 말고 `설정됨`, `미설정`, `교체 예정`, `삭제 예정` 상태로 보여준다.
- 실발송 ON은 수집 테스트와 전송 테스트가 끝난 뒤 확인하게 한다.
- 장기 과제로 DB 평문 저장을 secret ref 저장으로 바꾸되, 이번 UI 작업에서는 최소한 안내 문구와 상태 표현을 먼저 고친다.

### P1. 테스트 완료 상태와 실발송 ON이 안전한 완료 흐름으로 묶여 있지 않다

왜 중요한가:

- 설정이 저장돼도 실제 수집/전송이 되는지는 별도 확인이 필요하다.
- 실발송 기본 OFF는 안전하지만, "언제 ON 해도 되는지"가 화면에 명확하지 않다.
- dry-run, test-crawl, test-send 계열 액션은 존재하지만, 관리 탭의 고객 세팅 완료 조건으로 연결되어 보이지 않는다.
- 테스트 없이 실발송을 켜면 잘못된 채널이나 잘못된 센터로 메시지를 보낼 위험이 있다.

개선 방향:

- 고객별 세팅 상태를 `미완료`, `수집 테스트 통과`, `전송 테스트 통과`, `실발송 가능`으로 표시한다.
- `수집 테스트 통과 시각`, `전송 테스트 통과 시각`, `실발송 변경자`, `실발송 변경 시각`을 남긴다.
- 실발송 ON 버튼은 테스트가 끝난 뒤 노출하거나, 강한 확인 문구를 둔다.

### P1. 일부 HTMX 상태 피드백이 요청 성공 전에 성공처럼 보일 수 있다

왜 중요한가:

- 설정 화면은 "저장됨", "조회 완료", "연결됨" 같은 피드백을 믿고 다음 단계로 넘어간다.
- `_entity_admin.html:337`의 `loadDeliveryRules()`는 delivery-rule 목록과 option 요청을 보낸 직후 `조회 완료`를 표시한다.
- 네트워크 실패, 권한 실패, 서버 오류가 있어도 사용자는 완료로 오해할 수 있다.

개선 방향:

- HTMX 요청 전에는 `조회 중`과 `aria-busy="true"`를 표시한다.
- `htmx:afterSwap` 또는 성공 callback 이후에만 `조회 완료`를 표시한다.
- `htmx:responseError`에서는 같은 자리의 inline status에 실패 사유와 다시 시도 버튼을 보여준다.
- 마법사 단계도 같은 상태 규칙을 쓴다. 요청 중, 성공, 실패, 재시도 상태를 한 vocabulary로 통일한다.

### P2. 내부 용어가 화면에 많이 노출된다

왜 중요한가:

- 비개발 운영자는 `Tenant`, `external_id`, `webhook secret`, `SETUP_PENDING` 같은 용어에서 멈춘다.
- 화면을 이해하지 못하면 설정 실수가 늘어난다.

개선 방향:

- 화면 라벨을 쉬운 한국어로 바꾼다.
- 내부 값은 필요할 때만 보이게 한다.
- 예: `Tenant`는 `고객`, `external_id`는 `외부 관리 코드`, `sending_enabled`는 `실제 메시지 보내기`.

### P2. 작은 글자와 mono 중심 타이포그래피가 장시간 운영에 불리하다

왜 중요한가:

- 운영 화면은 빠르게 훑고 정확히 판단해야 한다.
- 작은 글자와 mono 본문은 한국어 가독성을 떨어뜨린다.

개선 방향:

- 본문/라벨 최소 14px, 입력값 14px 이상으로 맞춘다.
- 숫자, ID, 시간만 monospace로 남긴다.
- `dashboard.html`의 `0.68rem`부터 `0.76rem`까지 작은 텍스트 사용을 줄인다.

### P2. 좁은 화면에서 관리 폼 조작 부담이 크다

왜 중요한가:

- 모바일 CSS는 버튼 높이 44px를 보장하지만, 관리 탭의 fieldset 구조 자체는 줄어들지 않는다.
- 작은 글자, 많은 fieldset, 접힌 보조 영역, inline form이 합쳐지면 390px 화면에서 현재 위치와 다음 행동을 잃기 쉽다.
- 현장 운영자는 노트북이 아니라 좁은 화면으로 긴급 상태 확인이나 간단한 수정만 할 수도 있다.

개선 방향:

- 390px 폭에서는 각 단계가 한 줄 카드처럼 쌓이고, primary action이 fieldset 하단에 고정되게 한다.
- inline status가 버튼 옆으로 밀리지 않고 다음 줄 전체 폭으로 보이게 한다.
- 새 고객 세팅 마법사는 모바일에서 한 단계씩만 보여준다.
- 고급 설정은 모바일 기본 화면에서 닫아둔다.

### P2. 키보드 접근성과 폼 스타일 일관성이 아직 약하다

왜 중요한가:

- 모니터링 row는 클릭하면 drawer가 열리지만, row 자체는 `div onclick`이라 키보드 사용자는 같은 affordance를 바로 얻기 어렵다.
- tab UI는 `role="tablist"`와 `role="tab"`을 쓰지만, `aria-controls`, `tabpanel`, focus 이동이 빠져 보조기술 사용자에게 구조가 덜 명확하다.
- password input은 관리 폼 CSS 대상에서 빠져 text input과 다르게 보일 수 있다. 비밀값 입력은 신뢰가 중요한 영역이라 더 일관되어야 한다.

개선 방향:

- row 전체 클릭은 유지하되, row에 keyboard handler와 `tabindex="0"`을 추가하거나 row 안의 `상세` 버튼을 primary affordance로 더 명확히 한다.
- 모니터링/관리 탭에 `aria-controls`, panel `role="tabpanel"`, 선택 탭 focus 처리를 추가한다.
- `#view-manage input[type="password"]`도 text/number/select와 같은 스타일, focus, 크기 규칙을 탄다.
- 데스크톱에서도 주요 버튼의 최소 높이를 40-44px 수준으로 맞춘다.

## 6. 사용자별 위험

### 첫 사용자

위험:

- 어떤 순서로 설정해야 할지 모른다.
- 고객을 먼저 만들어야 하는지, 플랫폼 계정을 먼저 만들어야 하는지 헷갈린다.
- 텔레그램 토큰과 채널 생성, 고객별 텔레그램 설정의 차이를 이해하기 어렵다.

필요한 개선:

- 시작점 하나: `새 고객 세팅 시작`.
- 단계형 진행.
- 각 단계에 예시와 테스트 버튼 제공.

### 운영자

위험:

- 모니터링 중 작은 글자와 많은 배지가 피로를 준다.
- 문제 발생 시 "왜 멈췄는지"보다 내부 상태 코드가 먼저 보일 수 있다.

필요한 개선:

- 상태 문구를 원인과 다음 행동 중심으로 표시.
- 위험 상태 drawer에 `다음 조치`를 한 줄로 제공.

### 개발자/고급 관리자

위험:

- 고급 설정을 숨기면 디버깅이 어려울 수 있다.

필요한 개선:

- 고급 설정은 숨기되 쉽게 열 수 있게 한다.
- ID, raw 상태, webhook, registration code는 `고급` 모드에서만 표시한다.

## 7. 제안하는 목표 UX

### 7.1 새 고객 세팅 마법사

권장 단계:

1. 고객 선택 또는 생성
2. 플랫폼 선택: 배민 또는 쿠팡이츠
3. 플랫폼 계정 입력
4. 업체/센터 정보 입력
5. Chrome/Agent 연결 확인
6. 전송 채널 연결
7. 테스트 수집
8. 테스트 메시지 전송
9. 실발송 ON

각 단계의 원칙:

- 필요한 필드만 보여준다.
- 이미 아는 값은 자동으로 채운다.
- 실패하면 이유와 다음 행동을 바로 보여준다.
- 마지막에 체크리스트를 남긴다.

### 7.2 설정 상태 모델

고객별 상태:

| 상태 | 의미 | 다음 행동 |
| --- | --- | --- |
| 세팅 필요 | 필수 값이 부족함 | 마법사 이어서 진행 |
| 로그인 필요 | Chrome 세션 또는 Agent 인증 필요 | 로그인 안내 열기 |
| 수집 테스트 필요 | 설정은 됐지만 수집 확인 전 | 테스트 수집 실행 |
| 전송 테스트 필요 | 수집은 됐지만 채널 확인 전 | 테스트 메시지 전송 |
| 실발송 대기 | 테스트 완료, 실발송 OFF | 실발송 ON 검토 |
| 운영 중 | 수집과 전송 활성 | 모니터링 |

### 7.3 문구 방향

라벨은 사람이 하는 일을 기준으로 쓴다.

| 현재 문구 | 권장 문구 |
| --- | --- |
| Tenant | 고객 |
| external_id | 외부 관리 코드 |
| Platform Account | 배민/쿠팡 로그인 계정 |
| webhook secret | 텔레그램 webhook 보안키 |
| sending_enabled | 실제 메시지 보내기 |
| SETUP_PENDING | 세팅 중 |
| soft delete | 비활성화 |
| fail-closed | 기본 차단 |

## 8. 구현 시 주의

- 기존 검증 로직은 약화하지 않는다. 특히 쿠팡 기대 센터/상점명, CDP 로컬 주소, 텔레그램 필수값 검증은 유지한다.
- 비밀값을 문서, 로그, 응답, fixture에 남기지 않는다.
- 로컬 UI의 탭 기반 구조는 바로 제거하지 말고, 중앙 관리자 UI 세팅 마법사와 충돌하지 않게 역할을 줄인다.
- 다크 테마 자체는 유지 가능하지만, 설정 폼의 가독성은 높인다.
- 고급 설정을 숨겨도 검색이나 고급 모드로 접근 가능해야 한다.

## 9. 추천 작업 순서

1. `/admin` DB 연결 실패 시 사용자용 오류 화면 추가.
2. 관리 탭에 `새 고객 세팅 시작` 진입점과 세팅 체크리스트를 추가.
3. 용어와 UX copy를 먼저 정리해 내부 용어 노출을 줄인다.
4. 기본 설정과 고급 설정을 분리.
5. 전송 채널, 비밀값 상태, 실발송 게이트를 한 흐름에서 확인하게 정리.
6. 고객/계정/업체/채널/테스트/실발송 단계형 마법사 구현.
7. 글자 크기와 타이포그래피 정리.
8. 세팅 완료 상태와 테스트 결과를 고객별로 표시.

## 10. 최소 개선 단위

전체 마법사를 한 번에 만들면 템플릿과 라우트 변경이 커진다. 먼저 아래 3개를 작은 단위로 적용하는 편이 안전하다.

1. 장애 회복: `/admin` DB 실패 안내 화면과 테스트를 추가한다.
2. 빠른 이해: 관리 탭 상단에 `새 고객 세팅 순서` 체크리스트를 넣고, 기존 폼 위치로 이동하는 링크를 제공한다.
3. 실수 감소: 내부 용어와 작은 글자를 정리하고, 전송 채널/비밀값/실발송 상태를 같은 섹션에서 확인하게 한다.

이 3개만 해도 사용자는 "다음에 무엇을 해야 하는지"를 더 빨리 알 수 있다. 이후 실제 wizard를 HTMX fragment로 단계화하면 된다.
